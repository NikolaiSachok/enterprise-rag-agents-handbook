# Глоссарий

Единые определения терминов, на которые ссылаются страницы руководства. Каждый термин определён здесь один
раз. Список растёт по мере прохождения слоёв. Где за формулами и историей стоит канонический источник, в
конце определения стоит ссылка (↗ Wikipedia — для классики, ↗ arXiv — для приёмов из статей).

## Ingestion — чанкинг

**Chunk (чанк)** — фрагмент документа, единица индексации. Одновременно единица поиска и единица того, что
увидит модель.

**Chunk overlap (перекрытие)** — общий кусок текста у соседних чанков. Спасает факт, попавший на линию
разреза: он целиком уцелеет хотя бы в одном из соседей (если сам факт короче перекрытия). Обычно 10–20%
размера чанка.

**Recursive / structural chunking** — нарезка по естественным границам иерархически (разделы → абзацы →
предложения), чтобы границы чанков совпадали с границами мыслей. Выбор по умолчанию.

**Semantic chunking** — граница чанка ставится там, где резко падает смысловая близость соседних
предложений (смена темы). Дороже, но чанки получаются «про одно».

**Chunk metadata (метаданные чанка)** — данные, прикреплённые к чанку: источник, заголовок, путь по
разделам, дата, права доступа. Питают фильтрацию, цитирование и контроль доступа.

**Parent-document (small-to-big) retrieval** — искать по мелким точным чанкам, а в модель подавать больший
родительский фрагмент вокруг найденного. Разводит две роли чанка: поиск и контекст.

**Document parsing / layout-aware extraction (парсинг документов, извлечение с учётом разметки)** —
превращение сырого документа (PDF, скан, HTML) в структурный текст с сохранённым порядком чтения,
таблицами и иерархией заголовков. Layout-aware-парсер сперва распознаёт регионы страницы и порядок чтения,
а потом вытягивает текст — в отличие от плоского `extract_text`.

**OCR (оптическое распознавание символов)** — восстановление машиночитаемого текста из изображения или
скана, у которого нет текстового слоя. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Optical_character_recognition)

**Late chunking** — сперва прогнать весь длинный документ через эмбеддинг-модель, и только потом наложить
границы чанков и усреднить векторы токенов внутри каждого — так вектор чанка несёт контекст всего документа.
Нужна эмбеддинг-модель с длинным контекстом. ↗ [arXiv](https://arxiv.org/abs/2409.04701)

## Ingestion — эмбеддинги

**Embedding (эмбеддинг)** — вектор, представляющий текст в пространстве, где геометрическая близость
означает близость по смыслу.

**Embedding space (пространство эмбеддингов)** — векторное пространство, куда модель отображает тексты;
поиск сводится к нахождению ближайших точек к вектору запроса.

**Bi-encoder** — кодирует запрос и чанк по отдельности в два вектора и сравнивает их по расстоянию. Быстрый,
векторы чанков считаются один раз при индексации. Основа векторного поиска.

**Cross-encoder** — подаёт пару «запрос + чанк» в модель вместе и выдаёт одно число релевантности. Точнее
bi-encoder'а, но медленнее (score нельзя предпосчитать). Используется в реранкинге.

**Dimensionality (размерность)** — длина вектора эмбеддинга (например, 384 / 768 / 1536). Больше —
выразительнее, но дороже по памяти, скорости поиска и стоимости.

**Cosine similarity (косинусная близость)** — мера близости по углу между векторами; учитывает направление
и игнорирует длину. Метрика по умолчанию; у нормированных векторов совпадает с dot product.
↗ [Wikipedia](https://en.wikipedia.org/wiki/Cosine_similarity)

**Retrieval-optimized (asymmetric) embeddings** — модели, обученные на парах «запрос ↔ фрагмент (passage)»,
а не на общей похожести предложений. Часто ждут префикс `query:` / `passage:`.

**Multilingual embeddings** — эмбеддинг-модели, работающие в нескольких языках; критичны для
мультиязычного корпоративного контента.

**Self-hosted vs API embeddings** — выбор между открытой моделью на своей инфраструктуре (данные остаются в
периметре) и проприетарным API (проще, но данные уходят наружу и есть плата за вызовы).

**Embedding fine-tuning (тонкая настройка эмбеддингов)** — подгонка предобученной эмбеддинг-модели под
домен контрастным обучением на тройках (запрос, верный фрагмент, неверный фрагмент) с трудными негативами.
Требует полной переиндексации корпуса: запрос и документ обязаны использовать одну версию модели.

**Matryoshka Representation Learning (MRL)** — обучение эмбеддингов так, что информация уложена от грубого к
тонкому во вложенные префиксы, поэтому вектор можно усечь до меньшей размерности и он останется пригодным —
регулятор «размер/точность» без переэмбеддинга. ↗ [arXiv](https://arxiv.org/abs/2205.13147)

## Retrieval

**Dense retrieval (плотный поиск)** — поиск по векторам эмбеддингов; ловит смысл и синонимы, слеп к точным
токенам.

**Top-K** — число ближайших кандидатов, которые возвращает первая стадия поиска (обычно 50–100 до
реранкинга).

**Query transformation** — преобразование запроса перед поиском: переписывание, multi-query, HyDE.

**Multi-query** — генерация нескольких перефразировок запроса, поиск по каждой и объединение выдачи.

**HyDE (Hypothetical Document Embeddings)** — сгенерировать гипотетический ответ, заэмбеддить его и искать
по нему: он часто ближе к нужному чанку, чем короткий вопрос.
↗ [arXiv](https://arxiv.org/abs/2212.10496)

**BM25 / sparse retrieval** — классический лексический поиск по совпадению слов (частоты терминов). Ловит
точные токены, слеп к синонимам. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Okapi_BM25)

**Hybrid search (гибридный поиск)** — совместный запуск плотного и лексического поиска с объединением их
score. Закрывает слепые зоны каждого.

**Reciprocal Rank Fusion (RRF)** — способ объединить результаты нескольких поисков по позициям в их
ранжировках, без сведения разных шкал score.
↗ [SIGIR'09](https://cormack.uwaterloo.ca/cormacksigir09-rrf.pdf)

**Score fusion / score normalization (слияние по score / нормализация score)** — объединение ретриверов по
сырым score: привести каждый к общей шкале (min-max, z-оценка), затем взвешенно сложить. Сохраняет величину
score, но хрупко к выбросам и разным распределениям по запросам — это score-альтернатива ранговому RRF.

**Reranking (реранкинг)** — переоценка top-K кандидатов cross-encoder'ом и их пересортировка, чтобы лучшее
всплыло наверх. Вторая стадия, работает на точность.

**LLM reranker (LLM-реранкер)** — реранкинг через промпт к общей LLM, судящей релевантность: pointwise,
pairwise или listwise. Zero-shot и управляем инструкцией, но дорого, латентно и недетерминированно — в
отличие от специально обученного cross-encoder.

**Two-stage retrieval (двухстадийный поиск)** — дёшево и широко на полноту (bi-encoder / гибрид), затем
дорого и точно на точность (cross-encoder). Каноническая схема retrieval.

**Late interaction / ColBERT (позднее взаимодействие)** — закодировать запрос и документ в повекторные
представления на каждый токен, документ посчитать заранее, а на запросе оценивать через MaxSim: для каждого
токена запроса — максимум косинуса по токенам документа, сумма по токенам запроса. Держит потокенное
сопоставление уровня cross-encoder, оставаясь предподсчитываемым и годным для поиска по корпусу; цена —
хранилище, вектор на каждый токен. ↗ [arXiv](https://arxiv.org/abs/2004.12832)

**Multi-vector retrieval (мульти-вектор)** — представление чанка многими векторами (по одному на токен)
вместо одного; то, по чему ищет позднее взаимодействие.

**Contextual retrieval (контекстный поиск)** — приписать к каждому чанку короткую сгенерированную моделью
врезку, помещающую его в весь документ, ещё до эмбеддинга и BM25-индексации, чтобы чанк нёс контекст,
который голый чанк теряет; кэширование промпта делает это дёшево.
↗ [Anthropic](https://www.anthropic.com/news/contextual-retrieval)

**Query routing (маршрутизация запросов по индексам)** — заблаговременный выбор, где и как искать запрос: в
каком индексе или коллекции, идти ли в поиск вообще, плотный или гибрид, в каком срезе метаданных. Верх
воронки — неверный маршрут навсегда выбрасывает ответ из набора кандидатов.

**Metadata filtering (фильтрация по метаданным)** — сужение поиска по данным чанка: дата, отдел, тип,
язык.

**Pre-filter / post-filter (фильтр до / после поиска)** — когда предикат по метаданным или ACL применяется:
до векторного поиска (в кандидаты идут только прошедшие векторы — корректно и обязательно для ACL, но
избирательный фильтр конфликтует с ANN-индексом) или после (быстро, но избирательный фильтр может оставить меньше
K результатов, вплоть до пустой выдачи).

**Access control (ACL)** — отсечение чанков по правам доступа до выдачи, чтобы пользователь не получил то,
что ему не положено. Требование безопасности, а не опция.

**Recall@K / Precision@K** — метрики поиска: доля нужных документов, попавших в top-K (полнота), и доля
релевантных среди возвращённых (точность). ↗ [Wikipedia](https://en.wikipedia.org/wiki/Precision_and_recall)

**nDCG (normalized discounted cumulative gain)** — метрика ранжирования: учитывает не только попадание
релевантных документов, но и их позиции (выше — ценнее).
↗ [Wikipedia](https://en.wikipedia.org/wiki/Discounted_cumulative_gain)

**MRR (mean reciprocal rank)** — обратная позиция первого релевантного результата, усреднённая по запросам.
↗ [Wikipedia](https://en.wikipedia.org/wiki/Mean_reciprocal_rank)

## Generation

**Grounding (опора на контекст)** — привязка ответа к поданному контексту, а не к параметрической памяти
модели.

**Grounding instructions (grounding-инструкции)** — явное указание модели отвечать только по контексту и признаваться, когда ответа в
нём нет. Главный рычаг против галлюцинаций.

**Context packing (упаковка контекста)** — как найденные чанки собираются в промпт: разделение источников,
порядок, отбор нескольких лучших.

**Lost-in-the-middle** — модель хуже использует информацию, закопанную в середине длинного контекста, чем
ту, что стоит в начале и конце. ↗ [arXiv](https://arxiv.org/abs/2307.03172)

**Citations / attribution (цитирование)** — указание источника для каждого утверждения ответа; даёт
проверяемость и уменьшает число выдумок.

**Refusal / abstention (отказ)** — штатное «не знаю», когда контекста недостаточно; лучше уверенной ошибки.

**Faithfulness / groundedness (верность контексту)** — метрика того, насколько ответ действительно опирается
на поданные источники, а не на память модели.

**Parametric knowledge (параметрические знания)** — то, что модель выучила на обучении и хранит в весах; в
RAG его намеренно подавляют в пользу контекста.

**Hallucination (галлюцинация)** — уверенно поданный факт, которого нет в источниках или который неверен.

**Self-consistency (согласованность по выборке ответов)** — вместо единственного жадного прохода
chain-of-thought насэмплировать несколько путей рассуждения и взять финальный ответ большинством голосов;
применимо только там, где ответ — дискретное, голосуемое значение. ↗ [arXiv](https://arxiv.org/abs/2203.11171)

**Chain-of-verification / CoVe (цепочка проверок)** — цикл самопроверки: набросать ответ, спланировать
проверочные вопросы, ответить на них *независимо* от черновика (чтобы модель не подтвердила собственную
ошибку), затем переписать черновик по проверкам. ↗ [arXiv](https://arxiv.org/abs/2309.11495)

**Knowledge conflict (конфликт «контекст против памяти»)** — когда найденный контекст противоречит
параметрическому приору модели; модель не всегда держится контекста — особенно если приор укоренён, а
контекст выглядит неправдоподобно. Держится ли — меряет faithfulness.

**Answer-shaping (форма ответа)** — управление форматом, тоном и длиной ответа. Реальный рычаг качества, но
подчинённый опоре на контекст: форма не должна выкидывать цитату, оговорку или честный отказ.

## Evaluation

**Evaluation (оценка)** — измерение качества пайплайна метриками вместо «на глаз». Делает пайплайн
настраиваемым.

**Retrieval failure / generation failure (разделение провалов)** — диагностический костяк RAG: плохой ответ
бывает двух видов — *retrieval-провал* (нужного чанка не оказалось в выдаче) и *generation-провал* (чанк в
контексте был, но модель им не воспользовалась или переврала). Первый шаг отладки — понять, какой из двух
перед тобой.

**Golden set / golden dataset / ground truth (эталонный набор)** — примеры «вопрос + релевантные чанки / верный ответ»,
по которым считают метрики. Чистота важнее объёма.

**Answer relevance (релевантность ответа)** — reference-free generation-метрика: отвечает ли ответ на
заданный вопрос? LLM порождает N вопросов из ответа, а метрика берёт среднее косинусное сходство их
эмбеддингов с исходным вопросом — *(1/N) Σ cos(E_порожд_i, E_исход)*. Меряет совпадение по намерению, а не
фактическую верность. ↗ [arXiv](https://arxiv.org/abs/2309.15217)

**Correctness (корректность)** — совпадение ответа с эталонным по существу.

**LLM-as-a-judge** — оценка свободного текста другой LLM по заданным критериям или сравнением с эталоном;
масштабирует человеческую оценку на тысячи примеров.

**Judge bias (предвзятость судьи)** — систематические (не случайные) перекосы LLM-судьи, поэтому с ростом
числа примеров не усредняются: position bias (первый вариант — лечится перестановкой с требованием
устойчивости), verbosity bias (длиннее — лучше), self-preference / self-enhancement (свой стиль).
↗ [arXiv](https://arxiv.org/abs/2306.05685)

**Offline vs online eval** — оценка на golden set до деплоя (регрессии в CI) против измерения в проде
(обратная связь от пользователей, A/B).

**Regression eval** — прогон эталонного набора в CI, чтобы улучшение одного не сломало другое.

**A/B testing** — сравнение двух вариантов системы на живом трафике по метрикам.

**Faithfulness** — reference-free generation-метрика: ответ разбивают на атомарные утверждения, каждое сверяют
с найденным контекстом, берут долю подтверждённых — *faithfulness = подтверждённые утверждения / все
утверждения* (0–1). Меряет опору на контекст, а не правоту: утверждение, опертое на неверный контекст, всё
равно даёт 1,0. ↗ [arXiv](https://arxiv.org/abs/2309.15217)

**Context precision (точность контекста)** — retrieval-метрика с учётом ранга: стоят ли релевантные чанки
наверху выдачи? Взвешенное по рангам среднее Precision@k — перестановка одного нерелевантного чанка с позиции
2 на 1 роняет балл с ~1,0 до ~0,5. ↗ [arXiv](https://arxiv.org/abs/2309.15217)

**Context recall (полнота контекста)** — reference-based retrieval-метрика: вернул ли поиск всё, что нужно
эталонному ответу? Эталонный ответ разбивают на утверждения и берут долю, подтверждённую найденным
контекстом. Самый прямой замер retrieval-провала. ↗ [arXiv](https://arxiv.org/abs/2309.15217)

**Reference-free vs reference-based (без эталона / с эталоном)** — нужен ли метрике эталонный ответ. Без
эталона (faithfulness, релевантность ответа) считается по вопросу, контексту и ответу — годится на живом
трафике; с эталоном (context recall, корректность) нужен эталонный ответ.

**LLM-judge calibration (калибровка судьи)** — измерение согласия судьи с человеческими метками на отложенной
выборке до доверия в масштабе; сильные судьи достигают примерно человеческого уровня согласия (свыше 80%), но
не оракула. Перекалибровывать при дрейфе модели, корпуса или распределения вопросов.
↗ [arXiv](https://arxiv.org/abs/2306.05685)

**Pointwise vs pairwise (по одному ответу / попарное сравнение)** — два протокола судьи: pointwise выставляет
одному ответу абсолютный балл по критериям (дёшево, масштабируется, плывёт между прогонами); pairwise выбирает
лучший из двух (надёжнее для ранжирования, но O(n²) и уязвимее всех к position bias). Оценка с эталоном
(reference-guided) кладёт эталонный ответ в промпт судьи. ↗ [arXiv](https://arxiv.org/abs/2306.05685)

**Inter-annotator agreement (согласие между разметчиками, IAA)** — насколько независимые разметчики ставят
одинаковые метки; низкое согласие — сигнал править критерии оценки, а не переубеждать разметчика.

**Cohen's kappa (каппа Коэна)** — согласие двух разметчиков, поправленное на случайность:
*κ = (p_o − p_e) / (1 − p_e)*, где p_o — наблюдаемое, p_e — ожидаемое по случайности согласие.
↗ [Wikipedia](https://en.wikipedia.org/wiki/Cohen%27s_kappa)

**Fleiss' kappa (каппа Флейсса)** — обобщение каппы Коэна на более чем двух разметчиков.
↗ [Wikipedia](https://en.wikipedia.org/wiki/Fleiss%27_kappa)

**Active sampling / active learning (активный сэмплинг / активное обучение)** — тратить дефицитный бюджет
разметки там, где метки информативнее всего (где судья менее уверен, где судьи расходятся, где прод вскрыл
сбой), а не размечать вслепую.

## Guardrails

**Guardrails (ограничители)** — защитный слой на входе и выходе LLM-системы: против атак, утечек и вредного вывода.

**Prompt injection** — внедрение инструкций в текст, который читает модель, чтобы перебить системный промпт. Прямая (от пользователя) и косвенная (спрятана в retrieved-контенте — опасна для RAG).

**Spotlighting** — пометка недоверенного контента, чтобы модель видела его как данные, а не инструкции. Семейство из трёх приёмов на уровне промпта, нарастающих по силе и цене: **delimiting** (границы недоверенного текста спецтокенами), **datamarking** (маркер-символ вместо каждого пробела), **encoding** (base64 / ROT13 — текст больше не читается как инструкции). ↗ [arXiv](https://arxiv.org/abs/2403.14720)

**Instruction hierarchy (иерархия инструкций)** — приоритет источника, которому модель *обучают*: system / developer > user > tool / retrieved-контент. Указание с низкого уровня выполняется, только если согласовано с целью более высокого, и игнорируется, если конфликтует. ↗ [arXiv](https://arxiv.org/abs/2404.13208)

**PII redaction (маскирование PII)** — обнаружение и сокрытие персональных данных на входе и выходе; критично при использовании внешних API.

**Обратимое / необратимое маскирование** — как преобразуют найденную PII. Необратимое (удаление, замена, маскирование символами, хеширование) уничтожает оригинал; обратимое (шифрование) восстановимо по ключу. Обратимое маскирование даёт псевдонимизацию, а не анонимизацию, — а ключ становится обузой и мишенью.

**Input / output validation** — проверка входа (атаки, не по теме) и выхода (утечки, PII, нарушение политики).

**Content safety / moderation** — фильтрация вредного или недопустимого контента на входе и на выходе.

**Jailbreak** — обход встроенных ограничений модели (в отличие от injection, эксплуатирующего неотличимость инструкций от данных).

**Least privilege / tool allow-listing (принцип наименьших привилегий)** — ограничение набора инструментов и действий, доступных агенту, чтобы успешная инъекция мало что могла.

**Attack success rate (ASR)** — доля успешных атак на наборе; метрика качества guardrails.

**Defense-in-depth (эшелонированная защита)** — многослойная защита: ни один слой сам по себе не закрывает
всё, работают они вместе.

## Observability

**Observability (наблюдаемость)** — способность видеть, что живая система реально делает: отладить плохой
ответ, померить стоимость и латентность.

**Trace / span** — полная запись одного запроса сквозь пайплайн (trace) из шагов (span): запрос →
retrieval + score → промпт → вывод → шаги агента.

**RAG tracing** — трейсинг RAG-специфики: какие чанки достались и с каким score, финальный промпт, сырой
вывод.

**Cost per request / token accounting** — учёт стоимости и токенов на запрос; в LLM каждый вызов стоит
денег.

**Latency (p50 / p95)** — латентность по перцентилям; важнее всего шаги генерации и реранка.

**Three pillars (metrics / logs / traces)** — три столпа наблюдаемости; для LLM ключевой — traces.

**Feedback loop (observability → eval)** — прод-провалы и обратная связь от пользователей становятся новыми
кейсами golden set.

**Head-based sampling** — решение оставить или отбросить трейс в самом его начале, на корневом спане, по
доле от идентификатора трейса; дёшево и без состояния, но слепо к исходу запроса — предпочтительно сохранить
сбои нельзя.

**Tail-based sampling** — буферизация всех спанов трейса до его завершения и решение по всему трейсу
(латентность, ошибки, атрибуты); ловит интересные трейсы ценой состояния в памяти и балансировки по
`trace-id`. В OpenTelemetry Collector Contrib — процессор `tail_sampling`.
↗ [OpenTelemetry](https://opentelemetry.io/docs/concepts/sampling/)

**Priority / hybrid sampling (приоритетное сэмплирование)** — оставлять 100% того, что терять нельзя (ошибки,
превышения бюджета латентности, помеченные плохими ответы), а рутинные успешные запросы сэмплировать по
низкой базовой ставке; часто сперва head-сэмплер, затем tail.

**Message-content capture (захват содержимого сообщений)** — захват текста промпта и ответа в трейс; в
конвенциях OpenTelemetry GenAI по умолчанию выключен ради приватности и объёма, тогда как метаданные (модель,
число токенов, длительность) включены всегда.

**Retention (срок хранения)** — короткий TTL на спанах с содержимым (сырой текст быстро удаляется) и более
длинный на дешёвых метаданных; рычаг того, сколько живут чувствительные данные трейса.

**Golden signals (золотые сигналы)** — четвёрка сигналов из практики Google SRE — латентность, трафик,
ошибки, насыщение, — которые дашборд LLM-системы показывает рядом с полноценным графиком качества.
↗ [Google SRE](https://sre.google/sre-book/monitoring-distributed-systems/)

**SLI / SLO** — SLI (индикатор уровня сервиса) — измеряемая величина (доступность, p95-латентность, доля
прошедших по качеству); SLO (цель уровня сервиса) — планка на неё на горизонте. Для LLM-системы хотя бы один
SLI должен считаться онлайн-оценкой, а не только про аптайм.
↗ [Google SRE](https://sre.google/sre-book/service-level-objectives/)

**Error budget (бюджет ошибок)** — зазор между SLO и идеальными 100%: сколько сбоев тебе позволено потратить,
прежде чем цель нарушена.

**Burn rate (скорость сжигания)** — алертинг по тому, как быстро тратится бюджет ошибок (быстрое сжигание —
поднимать дежурного сейчас, медленный дрейф — предупреждать); альтернатива порогу на каждую метрику по
симптому.

**Alert fatigue (усталость от алертов)** — режим, когда алерт на каждую метрику топит настоящую регрессию в
шуме, и реальный инцидент остаётся непрочитанным.

**Разбор регрессий (regression triage)** — обнаружить статистически значимую просадку в ряду качества,
латентности или стоимости, а затем по спанам трейса возвести её к стадии и событию-изменению (деплой, смена
версии модели, переингест, дрейф входа).

**Cost attribution (атрибуция стоимости)** — разметка спанов признаками фича / арендатор / маршрут / модель,
чтобы счёт показывал, какой из них сжигает бюджет, а не только общий расход.

**Token accounting (поштучный учёт токенов)** — подсчёт входных плюс выходных токенов на запрос, каждый по
цене своей модели, через usage-атрибуты и метрики OpenTelemetry GenAI; основа бюджета стоимости.

**Latency budget (бюджет латентности)** — цель по p50 / p95 с раскладкой латентности по спанам (поиск,
реранкинг, генерация; TTFT против полного времени), чтобы превышение указывало на медленную стадию.

**Soft cap / hard cap (мягкий потолок / жёсткий потолок)** — политика бюджета: мягкий потолок предупреждает
(алерт, красный дашборд) и пропускает запрос; жёсткий потолок действует в среде выполнения (отклонить, спустить
на модель подешевле, урезать контекст).

## Agents — агентный RAG

**Agentic RAG** — RAG, в котором retrieval становится действием, выбираемым моделью в цикле, а не
фиксированным шагом конвейера. Управление ходом выполнения переходит от кода к модели.

**Agent loop (цикл агента)** — повторяющийся цикл «рассуждение → решение → действие → наблюдение», который
крутится, пока модель не решит, что готова отвечать.

**ReAct (Reasoning + Acting)** — паттерн «рассуждение → действие → наблюдение»: модель чередует шаги
рассуждения и действия (вызовы инструментов), подмешивая результат каждого действия обратно в контекст.
↗ [arXiv](https://arxiv.org/abs/2210.03629)

**Routing / query router (маршрутизация запросов по индексам)** — самый лёгкий уровень агентности: модель
делает один выбор, куда направить запрос (в какой индекс/инструмент или «retrieval не нужен»), а дальше
поток статичен. Не путать с model routing — маршрутизацией запросов между моделями (Часть III).

**Multi-hop retrieval (многошаговый поиск)** — ответ требует нескольких зависимых поисков, где следующий
запрос строится на результате предыдущего.

**Query planning (планирование запроса)** — заранее разложить сложный вопрос на подзапросы, прежде чем
искать.

**Self-correction / self-reflection (самокоррекция)** — агент оценивает промежуточную выдачу, замечает
«не то» и переформулирует запрос или ищет снова.

**Iterative retrieval (итеративный поиск)** — повторять поиск в цикле с уточнением, а не одним фиксированным
обращением.

**Self-RAG (само-рефлексивный RAG)** — RAG, в котором модель дообучена порождать особые токены рефлексии
прямо в потоке генерации: один тип решает, нужен ли поиск для этого куска ответа, остальные оценивают
релевантность найденного фрагмента, подтверждённость ответа фрагментом и его полезность. Решения об «искать
/ релевантно / подтверждено» встроены в генерацию, а не навешаны внешним каркасом.
↗ [arXiv](https://arxiv.org/abs/2310.11511)

**Corrective RAG, CRAG (корректирующий RAG)** — надстройка поверх любого RAG: лёгкий оценщик выдачи
(retrieval evaluator) выдаёт балл уверенности в трёх корзинах. Correct — оставить только релевантные
фрагменты найденного; Incorrect — выбросить выдачу и уйти в веб-поиск; Ambiguous — соединить оба пути.
Модель не трогает — работает plug-and-play. ↗ [arXiv](https://arxiv.org/abs/2401.15884)

**Adaptive RAG (адаптивный RAG)** — обученный классификатор предсказывает сложность запроса и направляет его
на самую дешёвую из достаточных стратегий: без поиска (ответ из параметрической памяти) / один заход в поиск
/ многошаговый итеративный поиск. Маршрутизатор из Части 1, сделанный под каждый запрос и обученный.
↗ [arXiv](https://arxiv.org/abs/2403.14403)

**Retrieval budget (бюджет поиска)** — жёсткие потолки на цикл поиска: макс. число заходов, поисков,
вытянутых токенов. Зеркалит бюджет шагов / бюджет токенов; последний рубеж, гарантирующий остановку цикла,
когда критерии достаточности не сработали.

**Sufficient context (достаточность контекста)** — критерий остановки поиска: собрано ли достаточно, чтобы
отвечать. Остановиться рано → ответ без опоры и галлюцинация; не остановиться → избыточный поиск, стоимость
и lost-in-the-middle. Токены подтверждённости / полезности Self-RAG — одна реализация, грейдер
релевантности — другая.

## Agents — инструменты

**Tool use / function calling (использование инструментов / вызов функций)** — общий механизм, которым модель
вызывает внешнюю функцию: модель порождает структурированное намерение, а исполняет его твой код. Retrieval —
частный случай.

**Tool definition (описание инструмента)** — имя, словесное описание и схема параметров (JSON Schema),
которые передаются модели: «меню» доступных инструментов. Описание работает как промпт — по нему модель
выбирает инструмент.

**Tool call (вызов инструмента)** — структурированный JSON (имя инструмента и аргументы), который модель
выдаёт вместо обычного текста или вместе с ним.

**Tool result (результат инструмента)** — результат исполнения инструмента, возвращённый модели в контекст
отдельным сообщением.

**Tool selection (выбор инструмента)** — решение модели, какой инструмент вызвать; частый источник ошибок при
большом или пересекающемся наборе инструментов.

**JSON Schema** — язык описания структуры и типов данных; задаёт допустимые параметры инструмента и сужает
то, что модель вправе выдать.

**Structured output (структурированный вывод)** — вывод модели в заданной машиночитаемой форме (JSON по
схеме), а не свободным текстом; основа надёжного вызова инструментов.

**Parallel tool calls (параллельные вызовы инструментов)** — несколько независимых вызовов инструментов,
которые модель выпускает за один ход; среда выполнения раскидывает их (fan-out — запускает разом) и собирает
результаты (fan-in). Законны, только когда вызовы не зависят друг от друга и не мешают друг другу;
управляются пофлагово у провайдеров (`disable_parallel_tool_use`, `parallel_tool_calls`).

**Constrained decoding (строгий (constrained) декодинг)** — принуждение схемы прямо при генерации: схему
компилируют в грамматику, и на каждом шаге маскируют всякий токен, который её нарушил бы, — результат валиден
по схеме по построению, а не по проверке постфактум.

**Strict mode / Structured Outputs (строгий режим)** — продуктовый переключатель (`strict: true`),
включающий строгий декодинг для аргументов инструмента; гарантирует правильно сформированные, валидные по
схеме аргументы (не верные по сути). Требует `additionalProperties: false` и всех свойств в `required`.

**Idempotency / idempotency key (идемпотентность / ключ идемпотентности)** — запись идемпотентна, когда
выполнить её дважды с одним входом — то же, что выполнить один раз; ключ идемпотентности позволяет серверу
отбрасывать повторы, так что повтор после таймаута безопасен. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Idempotence)

**Tool-RAG / dynamic tool loadout (динамический набор инструментов)** — поиск только тех инструментов, что
относятся к текущему запросу, и загрузка лишь их, вместо того чтобы возить весь каталог в каждом запросе;
RAG по меню инструментов. Режет токенную цену и ошибки выбора инструмента на больших наборах.

**Argument validation (валидация аргументов)** — проверка аргументов вызова перед исполнением на двух
уровнях: схемном (типы, enum, форматы) и семантическом (значения неверны по контексту — несуществующий id,
сумма сверх лимита).

**Retry budget (бюджет повторов)** — жёсткий потолок на число повторов, на вызов и на весь прогон; без него
детерминированно падающий вызов превращается в незавершающийся цикл повторов. Зеркалит бюджет шагов и бюджет
токенов.

## Agents — планирование и циклы

**Planning (планирование)** — как агент выстраивает последовательность шагов к цели: план может
фиксироваться наперёд или возникать по ходу цикла.

**Task decomposition (декомпозиция задачи)** — разбиение цели на подзадачи, которые агент берёт по одной;
бывает явной (записанный план / todo-список) и неявной (возникает по ходу рассуждения в цикле).

**Plan-and-execute (планирование-и-исполнение)** — стратегия, при которой весь план шагов строится наперёд,
а затем исполняется, с перепланированием при срыве шага; структурнее и дешевле ReAct, но жёстче.

**Re-planning (перепланирование)** — пересборка плана, когда шаг сорвался или наблюдение сломало исходный
план; обязательный механизм для plan-and-execute.

**Reflection / self-critique (рефлексия)** — выделенный шаг, на котором агент судит собственную траекторию
(«продвигаюсь ли я?») и решает остановиться, перепланировать или продолжить; главный рычаг против ухода от
цели и тихого зацикливания.

**Termination criterion (критерий остановки)** — заданное условие «готово», по которому цикл завершается;
часто реализуется как инструмент-сигнал «finish», который вызывает модель.

**Step budget / iteration limit (бюджет шагов / лимит итераций)** — жёсткий потолок на число шагов, вызовов,
токенов, стоимость или время; последний рубеж, гарантирующий остановку цикла в проде.

**Loop detection (обнаружение зацикливания)** — отслеживание повторения одного и того же действия (тот же
вызов, те же аргументы, тот же результат) с вмешательством, когда агент крутится вхолостую.

**Scratchpad / working memory (рабочая память)** — рабочее пространство, где агент держит только релевантное
для текущей траектории (промежуточные заметки, список сделанного), чтобы не раздувать контекст.

**Non-termination (незавершение цикла)** — фирменный сбой агентного цикла: он не останавливается,
застревает на повторяющемся действии или уходит от цели.

**Plan search (поиск по дереву/графу планов)** — планирование как перебор пространства кандидатных планов и
цепочек рассуждения вместо приверженности одному: породить несколько продолжений, выставить каждому балл
функцией ценности, разворачивать перспективные ветки с заглядыванием вперёд и пятиться из тупиков.

**Tree of Thoughts (ToT)** — обдуманный поиск по дереву промежуточных шагов рассуждения («мыслей»): модель
предлагает кандидатные мысли, сама оценивает каждое состояние баллом и обходит дерево (BFS/DFS) с
заглядыванием вперёд и возвратами, в отличие от единственной линейной цепочки chain-of-thought.
↗ [arXiv](https://arxiv.org/abs/2305.10601)

**Graph of Thoughts (GoT)** — обобщение ToT с дерева на произвольный граф: мысли-вершины и рёбра-зависимости
позволяют не только ветвить, но и сливать, агрегировать и дошлифовывать частичные решения.
↗ [arXiv](https://arxiv.org/abs/2308.09687)

**LATS (Language Agent Tree Search)** — поиск по дереву методом Монте-Карло по действиям агента (не только по
цепочке рассуждения): функцию ценности играет языковая модель, добавлены саморефлексия и обратная связь
среды; сводит воедино рассуждение, действие и планирование. ↗ [arXiv](https://arxiv.org/abs/2310.04406)

**Self-Refine** — итеративная доводка одной моделью без обучения: та же модель порождает ответ, критикует его
и переписывает по кругу, пока не станет достаточно хорошо. ↗ [arXiv](https://arxiv.org/abs/2303.17651)

**Reflexion** — «вербальное обучение с подкреплением»: после неудачной попытки агент пишет рефлексию на
естественном языке, кладёт её в буфер эпизодической памяти и вычитывает на следующей попытке — учится между
попытками без обновления весов. Имя фреймворка (Shinn и коллеги), не путать с рефлексией как общим приёмом.
↗ [arXiv](https://arxiv.org/abs/2303.11366)

**Episodic memory (эпизодическая память)** — хранилище прошлого опыта агента (что произошло, когда, чем
кончилось), которое переживает текущий контекст и достаётся по релевантности; в отличие от рабочей памяти —
черновика текущей задачи в контексте.

**Semantic memory (семантическая память)** — устойчивые факты, которые агент знает или выучил, обычно в базе
знаний или векторном хранилище; долговременная, в противоположность эфемерному scratchpad.

**Virtual context management (MemGPT)** — иерархия памяти по мотивам операционных систем: окно контекста как
«main context» (как оперативная память), внешнее хранилище как «external context» (как диск); модель
страницами подкачивает и выгружает данные вызовами инструментов и работает за пределами окна.
↗ [arXiv](https://arxiv.org/abs/2310.08560)

**Trajectory evaluation (оценка на уровне траектории)** — оценка всего пройденного агентом пути, а не только
финального ответа: результат (достигнута ли цель) и процесс (разумны ли шаги и вызовы инструментов), плюс
эффективность по шагам и завершаемость.

**pass^k** — доля задач, которые агент решает во всех k независимых попытках; метрика надёжности и
стабильности, вскрывающая разброс от прогона к прогону, который прячет одиночный pass@1.
↗ [arXiv](https://arxiv.org/abs/2406.12045)

## Agents — мультиагентные системы

**Multi-agent system (мультиагентная система)** — несколько специализированных агентов, работающих сообща,
вместо одного агента; мотивы — специализация, изоляция контекста, модульность, параллелизм.

**Orchestrator / supervisor (оркестратор / супервизор)** — ведущий агент, который декомпозирует задачу,
распределяет подзадачи по агентам-исполнителям и синтезирует их результаты; его «инструменты» — сами
субагенты.

**Worker / sub-agent (агент-исполнитель / субагент)** — специализированный агент, который берёт
распределённую ему подзадачу и возвращает результат.

**Handoff (передача управления)** — передача управления вместе с релевантным контекстом от одного агента
другому; сообщение передачи работает как промпт для принимающего агента.

**Agent chain (цепочка агентов)** — последовательная топология, в которой каждый агент преобразует вывод
предыдущего (например, автор → редактор → фактчекер).

**Critic / debate (критик / дебаты)** — топология, где агент-критик (или несколько независимых агентов)
оспаривает или сравнивает предложенные решения задачи, поднимая качество за счёт независимых точек зрения.

**FIPA ACL** — язык общения агентов, стандартизованный FIPA (2002): сообщение — это *перформатив* (речевой
акт — inform, request, propose, cfp…), оборачивающий поля (sender, receiver, content, ontology, protocol,
conversation-id). Многолетний предок нынешних схем сообщений между агентами.
↗ [Wikipedia](https://en.wikipedia.org/wiki/Foundation_for_Intelligent_Physical_Agents)

**Contract net protocol (протокол контрактных сетей)** — распределение задач через переговоры: менеджер
объявляет задачу, свободные исполнители присылают заявки, менеджер присуждает работу лучшей, исполнитель
возвращает результат (Reid G. Smith, 1980). Динамическое назначение роли, оформленное как обмен
сообщениями. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Contract_Net_Protocol)

**Blackboard (общая доска)** — архитектура координации через общую память: независимые специалисты
(источники знаний) читают и пишут одну глобальную структуру данных, а управляющий компонент решает, чья
очередь писать; агенты координируются через доску, а не окликая друг друга. Альтернатива попарной передаче
управления. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Blackboard_system)

**Multi-agent debate (мультиагентные дебаты)** — несколько экземпляров модели независимо предлагают ответ,
затем несколько раундов критикуют и правят чужие варианты, сходясь к более точному и связному результату,
чем даёт одиночный проход; протокольная форма топологии «критик / дебаты».
↗ [arXiv](https://arxiv.org/abs/2305.14325)

**Trajectory stitching (сшивка траектории)** — общий идентификатор разговора или задачи протягивается через
каждое межагентное сообщение, так что локальные трейсы агентов сшиваются в одно дерево «родитель — потомок»,
которое можно оценить целиком; предпосылка оценки команды.

## Agents — фреймворки оркестрации

**Orchestration framework (фреймворк оркестрации)** — библиотека, которая упаковывает цикл агента, обвязку
вызова инструментов, состояние, управление потоком и мультиагентную оркестрацию, чтобы не собирать это
вручную: LangChain, LangGraph, LlamaIndex, Microsoft Agent Framework (преемник Semantic Kernel и
AutoGen), CrewAI.

**Agent as a graph / state machine (агент как граф / конечный автомат)** — представление агента как узлов
(вызов модели / вызов инструмента / решение) и рёбер (управление потоком, включая циклы), чтобы цикл стал
наблюдаемым, возобновляемым и управляемым.

**Node / edge (узел / ребро)** — элементы графа: узел — шаг (вызов модели / вызов инструмента / решение),
ребро — управление потоком.

**Checkpointing (чекпоинты)** — сохранение состояния агента, чтобы прогон можно было приостановить,
возобновить и осмотреть.

**State graph (граф состояния, StateGraph)** — агент, описанный как общий типизированный объект состояния
плюс узлы и рёбра; конкретная форма «агента как графа» в LangGraph.

**Conditional edge (условное ребро)** — ребро графа, выбирающее следующий узел по текущему состоянию;
именно оно кодирует ветку цикла (есть вызов инструмента → узел `tools`; готово → END).

**Checkpointer (чекпойнтер)** — компонент, сохраняющий состояние графа на каждом супершаге с привязкой к
потоку, чтобы прогон можно было возобновить или отмотать назад (путешествие во времени).

**Checkpoint backend (бэкенд чекпоинтов)** — сменное хранилище под чекпойнтером (в оперативной памяти /
SQLite / Postgres / Redis); выбор между разработкой и продом.

**Thread (поток, thread_id)** — идентификатор, разводящий историю чекпоинтов одного разговора от другого.

**Durable execution (устойчивое исполнение)** — прогон, который после падения, перезапуска или долгой паузы
возобновляется с последнего успешного шага; строится на чекпойнтере, режимы `durability` (exit / async /
sync) меняют момент записи на скорость.

**Framework long-term memory / store (долгая память фреймворка, хранилище)** — сквозная по потокам
персистентная память с ключом-пространством имён, в отличие от короткого состояния чекпоинта в пределах
потока.

**Declarative vs imperative agent definition (декларативное / императивное описание агента)** — описание
агентов в конфигурации (YAML/JSONC, декларативные рабочие процессы) против сборки графа в коде
(add_node / add_edge).

**Human-in-the-loop (HITL) (человек-в-цикле)** — точка паузы, где человек одобряет или вмешивается, прежде
чем цикл продолжится; во фреймворке — полноправный узел-прерывание.

## Agents — MCP и протоколы агентов

**MCP (Model Context Protocol)** — открытый стандарт клиент-сервер (создан Anthropic в конце 2024,
с декабря 2025 — проект Agentic AI Foundation под Linux Foundation), связывающий
агентов с инструментами и данными; стандартизует инструменты, ресурсы и промпты. Превращает M×N штучных
интеграций в N+M. ↗ [modelcontextprotocol.io](https://modelcontextprotocol.io)

**MCP server (MCP-сервер)** — оборачивает инструмент или источник данных и единообразно выставляет его
возможности.

**MCP client (MCP-клиент)** — агент или приложение, которое подключается к MCP-серверам и потребляет их
возможности.

**MCP resources (ресурсы MCP)** — данные и контекст, которые выставляет MCP-сервер (у OpenAPI/CLI аналога
нет).

**MCP prompts (промпты MCP)** — переиспользуемые шаблоны, которые предлагает MCP-сервер.

**M×N integration problem (проблема M×N интеграций)** — M приложений × N инструментов = M×N штучных
коннекторов; стандарт сворачивает это в N+M.

**A2A (Agent2Agent)** — открытый стандарт (создан Google, объявлен в апреле 2025, с середины 2025 — проект
Linux Foundation) для связи агента с агентом: агент публикует Agent Card для обнаружения, а работу передают
как задачи-Task с сообщениями и артефактами поверх JSON-RPC. MCP — это агент ↔ инструменты, A2A — агент ↔
агент. ↗ [a2a-protocol.org](https://a2a-protocol.org)

**MCP host (MCP-хост)** — LLM-приложение (IDE, чат или среда выполнения агента), которое устанавливает
MCP-подключения и содержит один или несколько клиентов; каждый клиент держит соединение один-к-одному с
сервером. Хост, клиенты, серверы — три роли, а не две.

**Capability negotiation (согласование возможностей)** — рукопожатие инициализации, в котором клиент и сервер
сверяют версию протокола и объявляют, какие возможности каждый поддерживает, ещё до начала работы.

**Roots** — возможность клиента, которая задаёт серверу границы (файлы и URI), в пределах которых ему
дозволено работать; ограничение по наименьшим привилегиям, заданное клиентом, а не оставленное на
договорённость.

**Sampling** — возможность, через которую сервер просит модель клиента сгенерировать текст (своей модели у
сервера нет); требует явного согласия пользователя и ограничивает видимость промпта для сервера.

**Elicitation** — возможность, через которую сервер запрашивает у пользователя недостающие данные или
подтверждение по ходу работы, по структурированной схеме, которую рисует клиент.

**Streamable HTTP** — удалённый транспорт MCP (заменил HTTP+SSE в ревизии 2025-03-26); поддерживает несколько
клиентов и серверный стриминг и заставляет продумать аутентификацию и то, что открыто наружу в сеть.
↗ [modelcontextprotocol.io](https://modelcontextprotocol.io)

**MCP registry (реестр MCP-серверов)** — метареестр, хранящий метаданные серверов (не код и не бинарники),
чтобы клиенты могли обнаруживать существующие серверы; официальный запущен в превью 8 сентября 2025 года.
Наличие в реестре — не проверка. ↗ [registry.modelcontextprotocol.io](https://registry.modelcontextprotocol.io)

**Server discovery (обнаружение серверов)** — как клиент находит серверы для подключения: на уровне экосистемы
— через реестр (какие серверы существуют), при подключении — через рукопожатие возможностей (что предлагает
конкретный сервер).

**Tool poisoning** — косвенное внедрение инструкций через описание инструмента, которое модель читает как
промпт; самый опасный класс клиентских уязвимостей MCP.

**Rug pull** — сервер меняет поведение или описание инструмента уже после того, как ты его одобрил, так что
выданное при подключении доверие больше не соответствует тому, что инструмент делает. Противоядие —
фиксировать версию серверов и пересматривать их при изменении.

**Confused deputy** — привилегированный компонент, которого обманом заставляют злоупотребить своими правами в
пользу атакующего; классический риск при работе с OAuth-токенами удалённого MCP. Противоядие — наименьшие
привилегии и узкое ограничение прав токенов.

## Agents — реальные агенты (завершающий урок)

**Extended thinking (расширенное мышление)** — видимые блоки рассуждения, которые модель выдаёт перед
ответом; у Claude едут как `thinking`-блоки.

**Interleaved thinking (чередующееся мышление)** — рассуждение *между* вызовами инструментов, а не только
перед первым; у Claude включается на моделях с адаптивным мышлением.

**Reasoning effort (уровень рассуждения)** — управление глубиной рассуждения дискретным параметром
(`reasoning.effort` у OpenAI: `none`/`minimal`/`low`/`medium`/`high`/`xhigh`); сами токены рассуждения
непрозрачны и оплачиваются как выход.

**Thinking budget (бюджет размышления)** — числовой лимит на объём рассуждения на запрос (Gemini
`thinkingBudget`); в Gemini 3 сменяется дискретными уровнями `thinking_level`.

**Claude Code hooks (хуки)** — события жизненного цикла среды выполнения, из которых можно вызвать внешний
скрипт (`PreToolUse` умеет заблокировать вызов, `PostToolUse`, `Stop` и другие).

**ADK callbacks (колбэки)** — фиксированная матрица точек перехвата в ADK (`before`/`after` для агента,
модели и инструмента); возвращённый из колбэка объект закорачивает вызов.

**Permission modes (режимы разрешений)** — режимы, задающие, что агенту разрешено без подтверждения
(`default`/`acceptEdits`/`plan`/`bypassPermissions`…); проходятся в фиксированном порядке, где правило
`deny` блокирует даже под `bypassPermissions`.

## Production — сервинг

**Serving (сервинг)** — запуск модели или пайплайна как сетевого сервиса. Два смысла, которые нельзя
смешивать: сервинг приложения (RAG/агентный пайплайн, завёрнутый в API) и сервинг модели (собственный
инференс).

**Inference (инференс)** — вычисление моделью вывода по входу в проде, в противоположность обучению; то,
что либо покупают через API провайдера, либо запускают на собственных GPU.

**Inference server (инференс-сервер)** — специализированный сервер LLM-инференса на GPU: непрерывный
батчинг, управление KV-кэшем, OpenAI-совместимый API (vLLM, SGLang, Ollama).

**SSE (Server-Sent Events)** — односторонний поток событий поверх обычного HTTP; стандартный транспорт
стриминга токенов у LLM-провайдеров. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Server-sent_events)

**Time-to-first-token (TTFT)** — задержка до первого токена в потоке; метрика воспринимаемой скорости,
которую оптимизирует стриминг.

**Streaming (стриминг)** — отправка токенов пользователю по мере генерации, не дожидаясь полного ответа;
главный рычаг воспринимаемой задержки (TTFT).

**Continuous batching (непрерывный батчинг)** — планирование инференс-сервера, при котором запросы входят
в работающий батч и покидают его на гранулярности токена, а не по границам целого батча; главный рычаг
пропускной способности.

**PagedAttention** — управление KV-кэшем в vLLM по образцу виртуальной памяти: кэш разбит на страницы, что
практически убирает фрагментацию и поднимает пропускную способность.
↗ [arXiv](https://arxiv.org/abs/2309.06180)

**Cold start (холодный старт)** — задержка, прежде чем контейнер с моделью готов отвечать: веса грузятся в
видеопамять от десятков секунд до минут. Поэтому готовность ≠ запущенный процесс, а scale-to-zero имеет
цену.

**OpenAI-compatible API (OpenAI-совместимый API)** — стихийно сложившийся общий стандарт LLM-эндпоинтов: один
клиентский диалект говорит и с API провайдеров, и с self-hosted инференс-серверами; смена бэкенда почти
сводится к замене URL.

**ASGI workers (ASGI-воркеры, рабочие процессы)** — отдельные процессы ОС, у каждого своя копия ASGI-сервера
(uvicorn) и свой цикл событий; их поднимают, чтобы задействовать больше одного ядра. Параллелизм даёт цикл
событий, а не число воркеров: воркеры добавляют ядра и прикрывают процессорные вставки (сериализация,
токенизация, разбор JSON).

**uvloop** — быстрая реализация цикла событий на базе libuv, входит в `uvicorn[standard]`; подменяет
стандартный цикл asyncio и даёт прибавку в скорости без единой правки в коде.

**Threadpool offloading (пул потоков)** — вынос неизбежной синхронной работы из цикла событий в отдельный
поток (`run_in_threadpool`, `asyncio.to_thread`), чтобы блокирующий вызов не заморозил цикл, а с ним и все
параллельные запросы процесса.

**Backpressure (защита от переполнения)** — сознательное ограничение работы в полёте: семафор ставит предел на
число одновременных генераций, а очередь конечной глубины — на число ожидающих, так что сервис отказывает
избыточной нагрузке, а не берёт работу, которую не потянет.

**Load shedding (сброс нагрузки)** — быстрый отказ избыточным запросам, когда очередь полна (`429`/`503` с
`Retry-After`), вместо того чтобы их принять; отброшенный запрос клиент повторит, а расплавившийся для всех
сервис — нет.

**Admission control (контроль допуска)** — отклонение работы заранее, если к моменту исполнения она всё равно
превысит тайм-аут клиента, — чтобы не тратить слот GPU на ответ, которого уже никто не ждёт.

**Little's Law (закон Литтла)** — тождество теории очередей L = λW: средний параллелизм равен частоте прихода,
умноженной на время в системе. Поскольку у LLM-генерации W — десятки секунд, даже низкая частота запросов даёт
большой параллелизм. ↗ [Wikipedia](https://en.wikipedia.org/wiki/Little%27s_law)

**Iteration-level scheduling (планирование на уровне итерации)** — планирование инференс-сервера (непрерывный
батчинг), при котором новые запросы подсаживаются, а завершённые выселяются на каждом шаге декодирования, а не
по завершении целого статического батча; предложено в статье Orca (OSDI 2022).

**Prefill / decode (префилл / декодирование)** — две фазы генерации с противоположными узкими местами: префилл
прогоняет весь промпт за один проход и упирается в вычисления; декодирование выдаёт по одному токену за шаг,
заново перечитывая веса и KV-кэш, и упирается в пропускную способность памяти.

**Chunked prefill (префилл, разбитый на части)** — вплетение префилла длинного промпта в идущие декодирования в
одном шаге, чтобы один большой промпт не стопорил чужую генерацию токенов; чуть повышает p50 TTFT ради заметно
лучшего p95.

**Prefix caching (кэширование префикса)** — переиспользование KV-кэша общего начала промпта (например, общего
системного промпта) между запросами, минуя его повторный расчёт.

**Quantisation (квантизация)** — хранение весов (и, при желании, активаций) в пониженной точности — FP8, INT8
или INT4 через AWQ/GPTQ, ниже базовой FP16/BF16, — чтобы срезать память и поднять пропускную способность ценой
некоторой потери качества.

**KV-cache quantisation (квантизация KV-кэша)** — хранение самого KV-кэша в пониженной точности (например, FP8),
что примерно вдвое увеличивает число токенов в том же пуле — длиннее контексты или выше параллелизм.

**Tensor parallelism (тензорный параллелизм)** — разрезание весовых матриц каждого слоя между несколькими GPU;
после каждого слоя нужен all-reduce, чтобы собрать частичные результаты, поэтому режим требует много обмена и
быстрого интерконнекта (NVLink) внутри одной машины.

**Pipeline parallelism (конвейерный параллелизм)** — разрезание слоёв на ступени, каждая на своём GPU или своей
машине, с потоком микробатчей со ступени на ступень; обмена куда меньше, чем в тензорном, поэтому режим терпит
медленную сеть между машинами — ценой «пузыря» конвейера, пока он наполняется и опустошается.

**Data parallelism (параллелизм данных)** — целые копии модели за балансировщиком ради чистой пропускной
способности; применяют, когда модель и так помещается на один GPU, — в отличие от тензорного и конвейерного
параллелизма, которые нужны для моделей, что не влезают.

**MIG (Multi-Instance GPU)** — аппаратное разбиение A100/H100 на изолированные экземпляры, у каждого своя память
и своя изоляция сбоев.

**GPU time-slicing (нарезка GPU по времени)** — разделение одного GPU перемежением работы, без изоляции памяти и
сбоев; годится для тестового кластера, рискованно в проде, где сбой или всплеск памяти одного арендатора
достаёт остальных.

**KEDA** — событийный автоскейлер Kubernetes, масштабирующий нагрузки по внешним/пользовательским метрикам
(глубина очереди, токены в секунду, загрузка GPU), в отличие от штатного HPA, который видит только CPU и память.

**KServe** — слой модельного сервинга в Kubernetes (вместе с Knative), дающий автомасштабирование по запросам,
включая масштабирование в ноль и по числу одновременных запросов.

**Serverless GPU (serverless-GPU)** — GPU с посекундной оплатой, масштабированием в ноль и без забот о кластере
(Modal, RunPod, Replicate, Baseten, Cloud Run с GPU); главная его беда — налог холодного старта, который сбивают
снимком памяти и тёплыми пулами.

## Production — облачные платформы

**Managed endpoint (управляемый эндпоинт)** — модель, которую облачная платформа отдаёт как сервис внутри
твоего периметра IAM, биллинга и сети: ты её вызываешь, платформа её эксплуатирует.

**Model catalog (каталог моделей)** — набор собственных и сторонних моделей, которые платформа умеет
поднимать как управляемые эндпоинты (Foundry Models, каталог Bedrock, Model Garden).

**Data residency (резидентность данных)** — гарантия того, в каком регионе или геозоне обрабатываются
запросы; вместе с обязательством не обучать модели на твоих данных и частным подключением образует триаду
соответствия.

**Provisioned throughput (выделенная пропускная способность)** — зарезервированная под тебя ёмкость
модели с предсказуемой задержкой, которую покупают вместо общих on-demand токенов (Azure PTU, Vertex
Provisioned Throughput, уровень Reserved в Bedrock).

**Batch mode (пакетный режим)** — асинхронный тариф со скидкой для неинтерактивных нагрузок.

**Managed RAG (управляемый RAG)** — упакованный платформой конвейер от загрузки документов до поиска
(Bedrock Knowledge Bases, Foundry IQ поверх Azure AI Search, Vertex RAG Engine); скорость в обмен на
возможность тонкой настройки.

**Vendor lock-in (привязка к поставщику)** — зависимость, которую создают платформенные надстройки
(управляемый RAG, фирменные SDK), а не сам эндпоинт: он чаще всего OpenAI-совместимый.

**Fine-tuning (дообучение)** — продолжение обучения базовой модели на твоих данных, чтобы изменить её
поведение; на платформах это дообучение с учителем (SFT), методы на предпочтениях и RL (DPO, RFT),
дистилляция и продолженное предобучение. Дообучай под форму — стиль, схему, формат, — а не под меняющееся
знание: это работа RAG.

**LoRA / PEFT** — parameter-efficient-дообучение: правишь малый адаптер поверх замороженных весов, а не все
веса разом, — доля вычислений и хранения от полного дообучения.

**DPO (Direct Preference Optimization)** — дообучение на парах «предпочтительный / отвергнутый ответ», чтобы
выровнять выходы модели, без отдельной модели-награды.

**Reinforcement fine-tuning (RFT)** — дообучение по функции награды или модели-оценщику, которая оценивает
каждый ответ баллом, поощряя нужное поведение.

**Model distillation (дистилляция модели)** — обучение маленькой модели-ученика повторять выходы большой
модели-учителя: немного качества в обмен на кратно более дешёвое обслуживание.

**Continued pre-training (продолженное предобучение)** — расширение предобучения базовой модели на большом
объёме неразмеченных доменных данных, до любой узкозадачной настройки.

**Managed agent runtime (управляемая агентная среда)** — сервис платформы, который хостит сам агентный цикл и
добавляет персистентность сессий и памяти, шлюз инструментов и идентичности, наблюдаемость и масштабирование
в ноль (Bedrock AgentCore, Foundry Agent Service, Vertex Agent Engine) — дельта над агентом в своём
контейнере.

**FinOps** — дисциплина, связывающая инженерные решения с облачными расходами и сводящая их к обоснованному
числу юнит-экономики.

**Cost modelling (моделирование стоимости)** — оценка расходов нагрузки из её токенной формы и тарифных
рычагов платформы ещё до того, как ты на неё подписался.

**Unit economics (юнит-экономика)** — стоимость одной единицы ценности: запроса, активного пользователя,
фичи, — то число, которое оптимизирует FinOps.

**Committed-use discount (скидка за обязательство)** — сниженная эффективная цена в обмен на резервирование
выделенной ёмкости или обязательство по сроку потребления (резервирования Azure PTU, уровень Reserved в
Bedrock).

**Context caching (кэширование контекста)** — кэширование большого повторяемого контекста промпта на стороне
провайдера (пара к кэшированию промпта); тарифицируется как доля от свежего входа плюс плата за хранение.

**Cross-region egress (межрегиональный исходящий трафик)** — плата за передачу данных между регионами или
облаками; ценовая ось ползунка «резидентность или ёмкость».

**Multi-cloud gateway (мультиоблачный шлюз)** — маршрутизатор перед несколькими провайдерами или облаками с
единым OpenAI-совместимым проводом, ради переносимости, отказоустойчивости, маршрутизации по цене и
централизованного управления (LiteLLM, Portkey).

**Digital sovereignty (цифровой суверенитет)** — контроль над тем, кто может получить доступ к данным и
нагрузкам, эксплуатировать их и по закону затребовать, и под чьей юрисдикцией они находятся: операционный,
данных и программный суверенитет. Отличается от резидентности, которая лишь про то, где данные лежат.

**Sovereign cloud (суверенное облако)** — облако, построенное ради цифрового суверенитета: регионы под
европейским или национальным управлением, партнёрские «доверенные облака» или развёртывание, изолированное от
сети (AWS European Sovereign Cloud, Microsoft Sovereign Cloud, Google Distributed Cloud).

**Air-gapped (изолированный от сети)** — среда, полностью отрезанная от публичного интернета, для регулируемых
и оборонных нагрузок; фронтирные модели там часто отстают или отсутствуют.

## Production — экосистема инструментов

**Instrumentation (инструментирование)** — код и SDK-хуки, благодаря которым пайплайн испускает трейсы,
спаны и метрики; предпосылка наблюдаемости.

**OpenTelemetry GenAI conventions** — складывающийся вендор-нейтральный стандарт имён спанов и атрибутов
для LLM-вызовов (модель, токены, вызовы инструментов): инструментирование пишется один раз, экспорт — в
любую платформу. ↗ [GitHub](https://github.com/open-telemetry/semantic-conventions-genai)

**Safety classifier (классификатор безопасности)** — компактная специализированная модель, выставляющая
тексту на входе или выходе балл риска по категориям (Llama Guard, Granite Guardian); сочетается с
guardrails-фреймворками.

**Red-teaming** — намеренная атака на собственную систему ради измерения защиты (доля успешных атак,
ASR); в eval-продуктах существует как готовая функция.

## Production — LLMOps

**LLMOps** — операционная дисциплина LLM-приложений: деплой, мониторинг и стоимость систем, чьё поведение
живёт в промптах, версиях моделей, индексах и конфигурации, а не только в коде. MLOps, специализированный
под приложения на фундаментальных моделях.

**Canary release (канареечный релиз)** — раскатка новой версии (промпта, модели, индекса) на небольшую долю
живого трафика с наблюдением за метриками качества и стоимости: регрессия проявляется на доле пользователей
и дёшево откатывается. ↗ [Martin Fowler](https://martinfowler.com/bliki/CanaryRelease.html)

**Shadow deployment (теневой запуск)** — новый вариант работает на зеркальной копии трафика, его ответы
пользователи не видят; безопасное сравнение на реальных запросах.

**Prompt registry (реестр промптов)** — версионированное хранилище промптов, отвязанное от деплоев кода;
каждый ответ в проде привязывается к точной версии промпта.

**Model pinning (фиксация версии модели)** — закрепление точных снапшот-версий модели; обновление модели у
провайдера становится явным деплоем с прогоном eval.

**Model routing (маршрутизация запросов между моделями)** — отправка каждого запроса самой дешёвой модели, которая
с ним справится; не путать с маршрутизацией запросов по индексам (Часть I) и выбором инструмента (Часть II).

**Fallback (резервный маршрут)** — заранее настроенная альтернатива (другой регион, провайдер или модель) на
случай ошибок или превышения лимитов у основной.

**LLM gateway (LLM-шлюз)** — прослойка, централизующая доступ к моделям: один API, маршрутизация, резервные
маршруты, ключи, бюджеты (LiteLLM, OpenRouter).

**Prompt caching (кэширование промпта)** — кэширование повторяющегося префикса промпта на стороне
провайдера; закэшированные входные токены тарифицируются с большой скидкой. Промпт проектируют по схеме
«статичный префикс — переменный суффикс».

**Semantic caching (семантический кэш)** — возврат сохранённого ответа на почти совпадающий вопрос по
близости эмбеддингов; экономия в обмен на риск ложного попадания.

**Drift (дрейф)** — смещение входов, корпуса или модели провайдера под неизменной конфигурацией системы:
дрейф входного трафика, дрейф корпуса, дрейф модели у провайдера.
