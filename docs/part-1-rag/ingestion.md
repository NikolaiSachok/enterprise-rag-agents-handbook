# Ingestion (индексация)

Ingestion — офлайн-часть RAG: то, что происходит с документами **до** первого вопроса. Здесь
готовится всё, по чему потом пойдёт поиск. Порядок работ: парсинг документов → **чанкинг** →
метаданные → эмбеддинг в векторную базу.

Пока на этой странице разобраны **чанкинг** и **эмбеддинг-модели** — два столпа индексации. Осталось
добавить парсинг документов.

:::info[Как читать]

Каждая тема даётся сначала «как в индустрии вообще». Привязку к конкретному проекту (Strata-RAG)
выносим в отдельные case-study страницы — чтобы теория не смешивалась с деталями реализации.

:::

## Чанкинг (chunking)

### Зачем вообще резать документы

Наивный рецепт звучит так: «порежь документ на чанки». Но *почему* именно резать — и почему это первое
место, где всё ломается?

Резать заставляют два независимых ограничения:

1. **Эмбеддинг-модель сжимает целый кусок текста в один вектор.** Чем больше и разнороднее кусок, тем
   более усреднённым выходит этот вектор. Вектор абзаца про одну мысль — точный. Вектор целой
   40-страничной инструкции — размытое облако, которое плохо совпадает с *любым* конкретным вопросом.
2. **Чанк — это одновременно единица поиска и единица того, что увидит LLM.** Поиск идёт по чанкам, и
   найденные чанки же уходят в модель. Держи эту мысль в голове весь раздел: **у чанка сразу две роли, и они
   тянут требования к его размеру в разные стороны.**

### Ключевой компромисс: крупный чанк против мелкого

Сердце всей темы — одна таблица, из которой вырастает всё остальное:

| | Слишком **крупный** чанк | Слишком **мелкий** чанк |
|---|---|---|
| Что с эмбеддингом | Вектор размыт → плохо совпадает с конкретным запросом | Вектор точный |
| Что с контекстом для LLM | Много шума, нужное «тонет» (эффект *lost in the middle*), дорого по токенам | Теряется смысл — чанк не понять в отрыве |
| Типичная ошибка | Достали «вроде по теме», но нужный факт разбавлен | Достали фрагмент, но сам по себе он бессмыслен |

Классический пример мелкого чанка — потеря **референса**. Предложение из отчёта:

> «В третьем квартале он вырос на 20%.»

Отдельным чанком это мусор: *кто* «он»? *какого года* квартал? Эмбеддинг такого предложения ни с чем
осмысленно не совпадёт, а если и попадёт в выдачу — LLM не сможет им воспользоваться. Контекст
(«он» = выручка подразделения X за 2025) остался в соседнем абзаце, который в этот чанк не попал.

Вот почему подход «просто нарежь по N символов» проваливается — и почему появились стратегии умнее.

### Стратегии чанкинга (от простого к сложному)

1. **Fixed-size (фиксированный размер).** Режем каждые N токенов/символов. Просто, быстро,
   воспроизводимо. Минус: режет вслепую — посреди предложения, посреди таблицы. Базовый вариант, от
   которого отталкиваются остальные.
2. **Overlap (перекрытие).** Соседние чанки частично накладываются — «скользящее окно». Если факт попал
   на линию разреза, он целиком уцелеет хотя бы в одном из двух соседних чанков. Дёшево решает проблему
   разрезанных пополам фактов. Цена — дублирование текста. Применяется почти всегда вместе с любой
   стратегией; обычно перекрытие ~10–20% размера чанка.
3. **Recursive / structural (рекурсивный по структуре).** Режем не вслепую, а по естественным границам,
   иерархически: сначала по разделам, если кусок велик — по абзацам, затем по предложениям. Границы
   чанков совпадают с границами мыслей. **Выбор по умолчанию в индустрии** — компромисс «просто и почти
   всегда прилично».
4. **Semantic (семантический).** Идём по предложениям и смотрим на их эмбеддинги: пока соседние
   предложения близки по смыслу — это один чанк; резкое падение близости означает смену темы, то есть
   границу. Чанки получаются максимально «про одно». Минус — дороже (нужно эмбеддить уже на этапе
   нарезки) и не всегда оправдывает затраты.
5. **Document-structure-aware (по структуре документа).** Уважаем разметку исходника: заголовки,
   таблицы, блоки кода. Таблицу не режем построчно, код не рвём посреди функции, а **путь заголовков**
   («Глава 3 › Раздел 2 › Условия выплаты») прикрепляем к чанку как метаданные. Для корпоративных
   документов (регламенты, договоры, таблицы) это нередко решающий фактор качества.

:::tip[Универсального размера чанка не существует]

Он зависит от типа документа (плотный юридический текст ≠ болтливая переписка) и типа запросов
(точечный факт ≠ «объясни политику»). Поэтому размер чанка не угадывают, а **измеряют** — прогоняют
варианты через оценку поиска и смотрят на метрики. Отсюда мостик к слою
[Evaluation](./cross-cutting.md): чанкинг — не «настроил и забыл», а параметр, который тюнят по метрикам.

:::

### Метаданные чанка — где корпоративная специфика оборачивается качеством

Чанк — это не только текст. К нему прикрепляют **метаданные**: источник (файл/URL), заголовок, путь по
разделам, дату, версию и — критично для enterprise — **права доступа** (кто вправе это видеть). Три
причины заложить их уже здесь:

- **Фильтрация:** «ищи только в документах после 2024 года» / «только в разделе HR» — это фильтр по
  метаданным поверх векторного поиска.
- **Цитирование:** чтобы ответ ссылался на «Политику отпусков, раздел 2», эта ссылка должна ехать вместе
  с чанком с самого начала.
- **Access control:** в корпоративной системе сотрудник из маркетинга не должен получить в ответе кусок
  из зарплатной ведомости. Права проверяются на уровне чанка — и несут их именно метаданные.

**Вывод: метаданные закладываются на этапе чанкинга.** Не прикрепил тогда — потом взять неоткуда.

### Забегая вперёд: у чанка две роли — их можно развести

Вернёмся к тому, что чанк одновременно ищется и скармливается LLM. Продвинутая идея (подробно — в слое
[retrieval](./retrieval.md)): **это не обязательно один и тот же кусок текста.** Можно *искать* по мелким
точным чанкам (хороший эмбеддинг), а *в LLM подавать* более крупный родительский фрагмент вокруг
найденного (полный контекст). Это семейство приёмов — *parent-document / small-to-big retrieval*. Пока
достаточно запомнить, что развести две роли чанка возможно.

### Что забрать из раздела

- Чанк одновременно **ищется и читается моделью** — две роли, конфликт требований к размеру.
- Слишком крупный → размытый эмбеддинг и шум; слишком мелкий → потеря контекста и референсов.
- Стратегии: fixed → **+overlap** → **recursive (по умолчанию)** → semantic → structure-aware.
- **Метаданные** (источник, раздел, дата, права) закладываются здесь и питают фильтрацию, цитирование и
  access control.
- Размер чанка **измеряют метриками**, а не угадывают.

**Новые термины** → [Глоссарий](../glossary.md): chunk, chunk overlap, recursive/structural chunking,
semantic chunking, chunk metadata, parent-document (small-to-big) retrieval.

## Эмбеддинг-модели

Ты знаешь общую механику: чанк → вектор → векторная база. Здесь — нюансы, которые отделяют «посмотрел
ролик» от «понимаю, почему выбрали именно так».

### Что такое эмбеддинг на самом деле

Эмбеддинг — это вектор в пространстве, где **геометрическая близость означает близость по смыслу**.
Модель — обученная функция «текст → вектор»: тексты об одном и том же ложатся рядом, несвязанные — далеко.
Тогда поиск — это «найди векторы, ближайшие к вектору запроса».

Отсюда главное следствие: **качество поиска ограничено сверху качеством эмбеддингов**. Если вектор нужного
чанка не лёг рядом с вектором запроса, дальше по конвейеру это уже почти не спасти (гибридный поиск лишь
смягчит — об этом в слое retrieval). Поэтому выбор модели — не деталь, а фундамент.

### Bi-encoder против cross-encoder

Ключевое различие, из которого растут и векторный поиск, и реранкинг.

| | **Bi-encoder** | **Cross-encoder** |
|---|---|---|
| Как считает | Кодирует запрос и чанк по отдельности → два вектора → сравнивает по расстоянию | Подаёт пару (запрос + чанк) вместе → выдаёт одно число-релевантность |
| Точность | Ниже (видит тексты порознь) | Выше (видит взаимодействие слов) |
| Скорость | Быстро | Медленно |
| Предпосчёт | **Да** — векторы чанков считаются один раз при индексации | **Нет** — оценка зависит от пары |
| Где применяется | **Векторный поиск** по всей базе | **Реранкинг** топ-K |

Конфликт «точность против скорости» разрешают комбинацией: bi-encoder быстро достаёт топ-K из миллионов
(полнота), cross-encoder переоценивает эти K (точность). Шаг переоценки — это и есть **реранкинг**.
Подробнее в слое [retrieval](./retrieval.md).

### Как выбирают эмбеддинг-модель

- **Заточка под поиск (retrieval-optimized).** Не всякая модель обучена под поиск — нужны те, что учились
  на парах «запрос ↔ пассаж», а не на общей похожести предложений.
- **Размерность вектора** (384 / 768 / 1536+). Больше — выразительнее, но дороже: память, скорость поиска,
  стоимость. «Больше» ≠ «всегда лучше».
- **Язык и домен.** Модель, сильная на английском, может проседать на русском, юридическом тексте или
  коде. Для корпоративного мультиязычного контента это критично.
- **Максимальная длина входа** — какой чанк модель вообще способна принять. Прямая связь с чанкингом.
- **API против self-hosted.** Проприетарный API (OpenAI, Cohere, Voyage) — просто и сильно, но данные
  покидают периметр и есть плата за вызовы. Открытые self-hosted модели (E5, BGE, gte) — данные остаются
  внутри (для enterprise-приватности часто решающее), без поштучной платы, но инфру держишь сам.

### Метрика близости — коротко

**Косинусная близость** (угол между векторами; учитывает направление, игнорирует длину) — вариант по
умолчанию. Многие модели выдают нормированные векторы, где косинус ≈ dot product. Правило: бери метрику,
под которую модель обучена (указана в её карточке) — рассогласование бьёт по качеству.

### Две частые грабли

- **Одна модель для запросов и документов.** Заэмбеддил документы одной моделью, запросы — другой →
  векторы в разных пространствах → мусор. Следствие: смена модели означает переиндексацию всего корпуса.
- **Асимметрия query/passage.** Retrieval-модели часто ждут префикс (`query:` / `passage:`); перепутал —
  качество тихо просело.

### Что забрать из раздела

- **Bi-encoder** (быстро, предпосчитан, = векторный поиск) против **cross-encoder** (точно, по паре,
  = реранкинг).
- Качество поиска упирается в потолок качества эмбеддингов.
- Выбор по осям: retrieval-optimized · размерность · язык/домен · длина входа · **API vs self-hosted
  (приватность)**.
- Одна модель для запроса и документа; следи за метрикой и префиксами.

**Новые термины** → [Глоссарий](../glossary.md): embedding, embedding space, bi-encoder, cross-encoder,
dimensionality, cosine similarity, retrieval-optimized (asymmetric) embeddings, multilingual embeddings,
self-hosted vs API embeddings.
