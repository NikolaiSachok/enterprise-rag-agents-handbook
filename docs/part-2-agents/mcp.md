# MCP и протоколы агентов — единый разъём между агентом и всем остальным

В уроке про [использование инструментов](./tool-use.md) агент научился вызывать инструменты. Но каждая
интеграция была штучной: агент подключали к инструменту вручную, своим кусочком связующего кода — под этот
инструмент, под этот API, под этот формат ответа. Один агент, десяток инструментов — терпимо. А дальше это
начинает взрываться комбинаторно.

Пусть у тебя `M` приложений (агентов, IDE, чат-клиентов) и `N` инструментов (базы, API, файловые системы).
Если каждое приложение подключают к каждому инструменту отдельным коннектором, выходит `M × N` штучных
связок — и каждую надо написать, протестировать и потом чинить, когда API инструмента поменяется. Это
**проблема M×N интеграций**: рост не линейный, а как произведение. Ровно та же боль, из-за которой в
железе когда-то развёлся зоопарк несовместимых кабелей.

Лечит её стандарт. Оберни каждый инструмент **один раз** — в сервер, говорящий на общем протоколе, — и
научи каждое приложение быть клиентом этого протокола **один раз**. Теперь `M × N` схлопывается в `N + M`:
`N` серверов плюс `M` клиентов, и любое приложение говорит с любым инструментом без единой новой строчки
клея. Это и есть **MCP (Model Context Protocol)** — открытый протокол подключения модели к инструментам и
данным. Его собственная метафора — **USB-C для AI-инструментов**: один стандартный разъём вместо
фирменного кабеля под каждое устройство.

## Что такое MCP

**MCP** — открытый стандарт, который Anthropic представила в конце 2024 года. Устроен он по
клиент-серверной архитектуре, знакомой любому, кто писал сетевой код: одна сторона предоставляет
возможности, другая их потребляет.

- **MCP-сервер** оборачивает инструмент или источник данных и выставляет его возможности единообразно —
  так, чтобы их понял любой клиент, а не только тот, под кого сервер писали.
- **MCP-клиент** — это агент или приложение, которое подключается к серверам и потребляет эти возможности.
  Клиент не знает и не хочет знать, что там внутри сервера; ему хватает протокола.

Стандартизует MCP три вида возможностей — три примитива:

- **инструменты** (tools) — вызываемые функции. Тот самый инструмент из прошлого урока, только теперь его
  описание и вызов идут по общему протоколу, а не по самодельной схеме.
- **ресурсы** (resources) — данные и контекст, которые сервер отдаёт клиенту: содержимое файла, запись из
  базы, кусок документации. Не действие, а материал для работы.
- **промпты** (prompts) — готовые шаблоны, которые сервер предлагает клиенту: заготовленная формулировка
  задачи под конкретный сервер, которую пользователь может вызвать как есть.

Ходит всё это по одному из двух транспортов. Транспорт **stdio** — для локальных серверов, запущенных рядом с
клиентом как отдельный процесс: общаются через стандартный ввод-вывод. **HTTP/SSE** — для удалённых
серверов, живущих за сетью. Транспорт — деталь доставки; примитивы поверх него одни и те же.

## AI-дельта — инструменты отвязаны от агентов

Главное, что даёт MCP концептуально, — он **разрывает связь** между тем, кто строит инструменты, и тем, кто
строит агентов. Раньше инструмент и агент были склеены: функция, написанная под один агент, под другой
агент переписывалась заново вместе со всем клеем. С MCP ты пишешь сервер **один раз** — и им пользуется
любой MCP-клиент: другой агент, другая модель, другая IDE.

Это уже не про удобство одной интеграции, а про эффект экосистемы. Инструмент, собранный под одно
приложение, становится многоразовым по всем приложениям и моделям сразу. Тот, кто выставил свой сервис как
MCP-сервер, разом подключил его ко всему, что говорит на MCP; тот, кто написал MCP-клиента, разом получил
доступ ко всем существующим серверам. Клей больше не переписывают под каждую пару — в этом вся выгода
`N + M`.

## Чем MCP отличается от Swagger/OpenAPI и CLI `--help`

Тут живёт устойчивый миф, и его стоит разобрать прямо, потому что иначе ты унесёшь из урока неправильную
картинку. Миф звучит так: **«MCP — это Swagger для LLM, он просто добавляет описания к вызовам»**. Это
неверно, и неверно в самом корне.

Смотри, где ломается рассуждение. OpenAPI/Swagger **уже несут семантику**: в спецификации есть поля
`description` и `summary`, где по-человечески написано, что делает эндпоинт и что значит каждый параметр.
CLI-утилита несёт то же самое в тексте `--help`. И этой семантики достаточно, чтобы вести по ней модель:
можно взять OpenAPI-спецификацию, превратить эндпоинты в описания инструментов — и правило из прошлого
урока, «описание инструмента — это промпт», сработает здесь ровно так же. Модель прекрасно вызывает
функции по OpenAPI. Значит, отличие MCP **не в том, что у него есть смысл, а у них нет**. Смысл есть у всех
троих.

Настоящих отличий четыре, и они не про наличие описаний.

1. **Протокол потребления в среде выполнения против документации для разработчика.** MCP-клиент
   **обнаруживает** возможности сервера прямо во время работы агента и вызывает их единообразно. OpenAPI и
   `--help` описывают API для **человека**, который заранее прочитает документацию и напишет под неё
   клиентский код. Одно потребляет работающий агент на ходу; другое — документация, по которой инженер
   строит интеграцию заблаговременно.
2. **Примитивы, родные для LLM, помимо действий.** Ресурсы (контекст) и промпты (шаблоны) не имеют аналога
   ни в OpenAPI, ни в CLI — те описывают только вызываемые **действия**. MCP же стандартизует ещё и подачу
   контекста, и готовые шаблоны, а не одни функции.
3. **Единый клиент.** Любой MCP-клиент говорит с любым MCP-сервером — не нужен штучный клиент под каждый
   API или под каждый фреймворк. Вот здесь `N + M` из абстракции становится осязаемым: один раз написанный
   клиент открывает доступ ко всей экосистеме серверов.
4. **Сессия и двусторонность.** MCP — это сессия с состоянием, а не разовый запрос-ответ. Сервер может
   присылать обновления сам; а через **sampling** сервер вправе попросить модель клиента что-нибудь
   сгенерировать. Такой двусторонности у статической спецификации API нет по определению — она описывает
   вызовы в одну сторону.

Где же гипотеза «это просто инструменты с хорошими описаниями» всё-таки права? Она права **на практике,
как соглашение, а не как возможность протокола**. MCP-серверы пишут сразу под агента: с оглядкой на ту
самую дисциплину «описание — это промпт» и на «мало инструментов, и они не пересекаются» из прошлого урока.
Поэтому типичный MCP-сервер — отобранный, сфокусированный набор инструментов, который модель читает легче,
чем сырой авто-сгенерированный Swagger-дамп на двести эндпоинтов. Но это отличие **практики
проектирования**, а не семантическая способность, которой у OpenAPI не хватает. Swagger отлично несёт
смысл; просто MCP-серверы, как правило, с самого начала пишут так, чтобы их потреблял не человек, а модель.
Не путай удачное соглашение с новой возможностью.

## MCP против A2A — агент-к-инструментам против агент-к-агенту

MCP стандартизует связь **агент ↔ инструмент/данные**: как агент дотягивается до внешнего мира. Но в
[мультиагентных системах](./multi-agent.md) есть отдельная, параллельная нужда — связь **агент ↔ агент**:
как один агент передаёт задачу другому и получает результат. Это не то же самое, и MCP её не закрывает.

Под неё складывается свой стандарт — **A2A (Agent-to-Agent)**, предложенный Google, и он не единственный
претендент. Пространство агентных протоколов сейчас движется быстро, имена приходят и уходят, поэтому
запоминать надо не список кандидатов, а разделительную черту, которая переживёт конкретные названия:
**MCP — это агент-к-инструментам и контексту; A2A — агент-к-агенту**. Держи в голове эту ось, и любой новый
протокол ты сразу разложишь по своей полке.

## Безопасность — новая поверхность атаки

Каждый MCP-сервер, который ты подключаешь, — это **новая поверхность атаки**. Вредоносный или взломанный
сервер способен на три неприятные вещи разом. Он может подсунуть косвенное **внедрение инструкций в текст
(prompt injection)** — спрятать команду в своих ресурсах или в результате вызова, который модель прочитает
как указание. Он может утащить данные наружу. И он может выйти за выданные ему права, дотянувшись до
большего, чем ты собирался ему доверить.

Защита — та же дисциплина, что и в уроке про инструменты, только теперь распространённая на чужие серверы:

- **Принцип наименьших привилегий (least privilege)** и ограниченный набор инструментов: чем меньше сервер
  может, тем меньше цена его компрометации.
- **Доверять только проверенным серверам.** Подключённый MCP-сервер исполняется в контуре твоего агента;
  случайный сервер из сети — это чужой код у тебя внутри.
- **Требовать подтверждения человека** на чувствительные действия — то самое право вето, которое ты уже
  видел в других уроках.

И сквозная нить безопасности из Части I тянется сюда без изменений: **иерархия инструкций** и
**spotlighting** распространяются на ресурсы и результаты MCP-сервера ровно так же, как на любой внешний
текст. Контент, пришедший от сервера, — это недоверенный ввод, а не доверенные инструкции. Оборачивая
инструмент в MCP, ты не выносишь его за периметр безопасности — ты добавляешь этому периметру ещё одну
границу, которую надо стеречь.

## Что забрать из урока

- **Проблема M×N интеграций**: `M` приложений × `N` инструментов = штучные коннекторы, растущие как
  произведение. Стандарт схлопывает это в `N + M` — обернул сервер один раз, написал клиента один раз, и
  любое приложение говорит с любым инструментом. MCP — **USB-C для AI-инструментов**.
- **MCP** — открытый клиент-серверный стандарт (Anthropic, конец 2024). **MCP-сервер** выставляет
  возможности, **MCP-клиент** их потребляет. Стандартизует три примитива: **инструменты**, **ресурсы**,
  **промпты**. Транспорт — stdio локально, HTTP/SSE удалённо.
- AI-дельта: MCP **отвязывает инструменты от агентов** — сервер пишут один раз, им пользуется любой клиент.
  Это эффект экосистемы, а не удобство одной интеграции.
- Миф «MCP — это Swagger для LLM» неверен, но **не потому, что у Swagger нет семантики** — она есть, и по
  OpenAPI можно вести модель. Настоящие отличия: потребление в среде выполнения против документации для
  разработчика; примитивы за пределами действий (ресурсы и промпты); единый клиент под все серверы; сессия
  с состоянием и двусторонней связью (вплоть до sampling). «Лучше описанные инструменты» — это соглашение о практике, а не возможность,
  которой OpenAPI лишён.
- **MCP = агент-к-инструментам; A2A (Agent-to-Agent) = агент-к-агенту.** Запоминай ось, а не список
  протоколов, — пространство быстро меняется.
- MCP-сервер — **новая поверхность атаки**: косвенный prompt injection, утечка данных, выход за права.
  Защита прежняя — **принцип наименьших привилегий**, только проверенные серверы, подтверждение человека на
  опасное; иерархия инструкций и spotlighting из Части I накрывают и содержимое серверов.
