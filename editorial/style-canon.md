# Style canon — термины, резервации и конвенции корпуса

Канон стиля и терминологии Enterprise RAG & Agents Handbook. Это **нормативный** документ: новые страницы
пишутся по нему, а найденные в корпусе отклонения правятся под него. Канон живёт вместе с текстом —
обновление канона едет **в том же PR**, что и страница, ради которой оно понадобилось. Правила процесса
(редакционные проходы, чек-листы) живут в `CLAUDE.md`; здесь — только языковые и структурные нормы.

Языковая пара: **RU — основной язык при написании** (пишется нативно, не переводом). После перехода на
EN-canonical английский стал **локалью по умолчанию** и лежит в `docs/`, а русский — параллельная локаль в
`i18n/ru/...` (отдаётся на `/ru/`).

## 1. Реестр терминов

### 1.1. Термины, которые остаются английскими (kept-EN)

Без устоявшегося русского эквивалента; пишутся латиницей, при первом упоминании на странице — русское
пояснение в скобках, дальше без него:

*grounding, bi-encoder, cross-encoder, prompt injection, jailbreak, spotlighting, HyDE, BM25, ReAct,
plan-and-execute, faithfulness, top-K, late interaction, guardrails, eval/evaluation, observability, golden
set, LLM-as-a-judge, retrieval failure / generation failure, JSON Schema, sampling, elicitation, stdio,
streamable HTTP, tool poisoning, span → см. §1.2 (спан), trace → см. §1.2 (трейс),
scale-to-zero (масштабирование в ноль), on-demand (тариф «плати за токены»; глосс «по требованию»),
red-teaming (наступательное тестирование защиты — атакуешь собственную систему), source-available (код
открыт для чтения, но лицензия не open source), forward pass (прямой проход), liveness probe / readiness
probe, SLA, IAM, PTU (provisioned throughput units), SSE (Server-Sent Events), PagedAttention,
KV-кэш (латинское KV + кириллическое «кэш» через дефис), few-shot, Self-RAG, corrective RAG (CRAG),
adaptive RAG, Tree of Thoughts (ToT), Graph of Thoughts (GoT), LATS, Monte Carlo Tree Search (MCTS),
Self-Refine, Reflexion, MemGPT, chain-of-thought, τ-bench, pass^k / pass@1.*

**plan-and-execute** — имя стратегии, остаётся латиницей; русский глосс первого упоминания —
«plan-and-execute (планирование-и-исполнение)». Не изобретать голую кириллическую форму.

**Self-RAG / corrective RAG (CRAG) / adaptive RAG** — имена опубликованных архитектур agentic RAG, остаются
латиницей в обоих языках; русский глосс первого упоминания — «Self-RAG (само-рефлексивный RAG)», «corrective
RAG / CRAG (корректирующий RAG)», «adaptive RAG (адаптивный RAG)». Никогда не транслитерировать. Self-RAG —
всегда через дефис с заглавной «S»; в связной прозе прилагательные corrective / adaptive допустимы строчными.
**adaptive RAG — без дефиса** (не «Adaptive-RAG», хотя статья названа с дефисом): это прилагательное +
RAG, параллельно «corrective RAG», а не жёсткое имя-моноблок вроде Self-RAG. Дефисная форма «Adaptive-RAG»
отвергнута ради единообразия внутри корпуса.

**Tree of Thoughts (ToT) / Graph of Thoughts (GoT) / LATS / Self-Refine / MemGPT** — имена опубликованных
приёмов и систем планирования, рефлексии и памяти; остаются латиницей в обоих языках, не транслитерируются.
Русские глоссы первого упоминания: ToT — «поиск по дереву мыслей», GoT — «граф мыслей», LATS — «поиск по
дереву действий», MemGPT — «виртуальное управление контекстом». **MCTS** (Monte Carlo Tree Search) — «поиск
по дереву методом Монте-Карло».

**Reflexion ≠ рефлексия — ложный друг, развести жёстко.** `Reflexion` (с заглавной «R», латиницей) — имя
фреймворка (Shinn и др., 2023), НЕ транслитерируется и НЕ смешивается с «рефлексией» (§1.2) — общим приёмом.
Строчная кириллическая «рефлексия» = понятие; латинская `Reflexion` = конкретный фреймворк. На странице, где
встречаются оба, различие проговаривается явно (прецедент — planning-loops/deep-dive).

**pass^k / pass@1 / pass^8** — нотация метрик надёжности агента: строчное `pass`, показатель степени/индекс
как есть; латиницей в обоих языках (§3). pass^k — доля задач, решённых во всех k независимых попытках.

**streamable HTTP** — каноническое имя удалённого транспорта MCP. Ранний транспорт HTTP+SSE признан устаревшим
в ревизии спецификации MCP 2025-03-26 — «HTTP/SSE» в тексте корпуса не писать.

**sampling / elicitation** — возможности MCP, остаются латиницей. Канонические глоссы: sampling — «сервер
просит модель клиента сгенерировать текст»; elicitation — «сервер запрашивает у пользователя недостающие
данные». Правило первого упоминания: определяющее предложение, стоящее вплотную к термину, само выполняет
роль глосса — отдельный скобочный мост тогда не нужен.

**observability / наблюдаемость** и **evaluation (eval) / оценка** — двойное каноническое написание:
английская и русская формы равноправны и выбираются по ритму фразы; запрещены только транслитерации
(«обсервабилити», «эвал»).

Канонические имена слоёв — как есть, с заглавной: **Ingestion, Retrieval, Generation, Agentic RAG**.
Кодовые аббревиатуры — как есть: **ACL, PII, ASR, MCP, KB, RRF, MRR, nDCG, HITL, A2A**.

**Имена фреймворков и продуктов** — собственные имена, остаются латиницей в обоих языках, никогда не
транслитерируются: **LangChain, LangGraph, LlamaIndex, Microsoft Agent Framework, Semantic Kernel,
AutoGen, CrewAI, LangSmith, OpenAPI, Swagger, FastAPI, Pydantic, uvicorn, Docker, Kubernetes, vLLM, SGLang,
Ollama, Hugging Face (TGI, HF_HOME), Microsoft Foundry (экс-Azure AI Foundry), AWS Bedrock, Vertex AI /
Gemini Enterprise Agent Platform (GEAP), Model Garden, Amazon Nova, Azure AI Search, Foundry IQ, Ragas,
DeepEval, promptfoo, Langfuse, Arize Phoenix, OpenTelemetry (OTel), OpenInference, NVIDIA NeMo Guardrails,
Llama Guard, Granite Guardian, Model Armor, LiteLLM, OpenRouter, Colang, Responses API, Conversations API,
Agents SDK (OpenAI), Claude Agent SDK, ADK (Agent Development Kit), McpToolset, SequentialAgent, ParallelAgent,
LoopAgent, AgentTool, Moderation API**.

**FIPA ACL / blackboard / Agent Card / A2A** — имена стандартов и артефактов мультиагентной координации,
остаются латиницей в обоих языках, не транслитерируются. Русские глоссы первого упоминания: FIPA ACL —
«(язык общения агентов, стандарт FIPA)»; blackboard — «(общая доска / общая память команды)»; Agent Card —
«(карточка агента — его возможности и адрес для обнаружения)». **contract net protocol** — в теле RU идёт
нативная форма «протокол контрактных сетей» (§1.2), EN-оригинал мостом. **A2A** — каноническое имя
«A2A (Agent2Agent)» (бренд Google); дефисная форма «Agent-to-Agent» — только там, где раскрывается акроним
под ось «агент-к-агенту» (§3). «блэкборд» ✗.

## 2. Семантические резервации

Одно слово — одно значение на весь корпус. Нарушение резервации — дефект, даже если фраза сама по себе
грамотна.

- **«отказ» = refusal/abstention** — и только это: штатное «не знаю» как поведение системы (фича).
  Глагол «отказывать» в guardrails (система отказывает во вредном запросе) — та же семья «система говорит
  нет», допустим. Для *failure* слово «отказ» **запрещено** (в т.ч. «режим отказа», хотя в общей инженерной
  лексике это нормальный термин — здесь он сталкивается с резервацией книги). Незавершение агентного цикла
  (non-termination) называем «провал» / «сбой» / «цикл не останавливается» / «зацикливание» / «уход от
  цели» / «незавершение цикла» / «бесконечный цикл повторов» (детерминированно падающий вызов без бюджета
  повторов) — **никогда «отказ»**.
- **«провал» = failure.** Канонические термины — **retrieval failure / generation failure** (kept-EN);
  русский глосс — «провал», мостовая форма первого упоминания — «retrieval-провал (retrieval failure)».
  Имя сквозной рамки Части I — **«разделение провалов»** (допустимый вариант в тексте уроков —
  «декомпозиция провалов»).
- **Смысл, а не строка.** Строки реестра и резервации связывают **терминологический смысл**, а не сырое
  слово. Когда английское слово термина употреблено как обычное слово, перевод следует естественной
  идиоме, а не глоссу термина: *the core failure of this approach* — про фундаментальную проблему
  подхода → «главная проблема», не «главный провал».
  Уточнение по «провалу»: слово работает как **имя класса сбоя внутри таксономии** (retrieval-провал,
  разделение провалов, прод-провалы) — но не как **приписанное свойство** слоя или подхода: «у слоя есть
  провал» ✗ → «у слоя есть фирменная беда / слабое место / главная проблема»; в определении класса
  предпочитай «сбой» («non-termination — фирменный сбой агентного цикла»).
  **Дискриминация «провал» vs «ошибка/сбой» — порог перед словом.** «Провал» — это ИМЯ ТАКСОНОМИЧЕСКОЙ
  КАТЕГОРИИ отказа: «какой этап конвейера виноват» (retrieval-провал, generation-провал, разделение
  провалов). Слово про **атрибуцию / локализацию** вины, а не про сам факт, что что-то отказало.
  **Пороговый вопрос перед тем, как написать слово:** «Это ИМЯ КЛАССА отказа (какой этап виноват) или
  ЕДИНИЧНЫЙ ИНЦИДЕНТ / происшествие?» → категория-этап → «провал»; единичный отказ / ошибка / происшествие →
  «ошибка» / «сбой», **никогда «провал»**. Зачем разводить: «ошибка генерации» двусмысленна (единичный
  инцидент против «виновата генерация»); отдельное слово «провал» под категорию снимает эту двусмысленность.
  Один и тот же английский *failure* приземляется по-разному по смыслу: generation failure = категория →
  «провал»; validation failure = инцидент → «ошибка». Отсюда рабочий пример запрещённой рамки:
  «провал валидации» ✗ → **«ошибка валидации»** (единичное происшествие проверки; соседи по таксономии
  раздела «Ошибки и повторы» тоже событийные — «кривые аргументы», «исключение», «таймаут»; для общего
  «вызов детерминированно упал» подходит и «сбой»). Так же «детерминированный провал» вызова ✗ →
  «детерминированный сбой» (вызов, падающий детерминированно, — это сбой-инцидент; зеркалит «преходящий
  сбой»). «Провал» оставляем за **именами категорий** (retrieval-/generation-провал, разделение провалов,
  «Новые провалы», callback «тот самый провал из Части 1») и за **non-termination** («незавершение — это
  провал, сбой цикла»).
- **«сбой» / «ошибка»** — обычные слова для технических сбоев и ошибок вне таксономии провалов; ими не
  называют ни refusal, ни retrieval/generation failure.
- **«оценка» = evaluation** (дисциплина/процесс измерения качества). Смыслы рядом:
  1. процесс/дисциплина — «оценка пайплайна», «оценка поиска», «человеческая оценка», «критерии оценки»
     (rubric, §1.2) ✓;
  2. акт оценивания — глагольная семья («судья оценивает ответ», «cross-encoder заново оценивает
     кандидата») ✓ как обычное слово;
  3. **число**, которое выдаёт модель или судья, — **score** или **балл**, никогда «оценка»:
     «оценка зависит от пары» ✗ → «балл зависит от пары» (иначе читается «мнение/отметка», а не число);
  4. бытовую прикидку («по оценкам», estimate) рядом с eval-контекстом не писать — «примерно»,
     «по прикидкам».
- **«релевантность ответа» = answer relevance** (generation-метрика); не сокращать до «релевантность»
  там, где рядом обсуждается релевантность выдачи.
- **«выдача»** — то, что вернул поиск (results); не «результаты поиска» в терминологическом смысле и не
  «ответ» (ответ — это то, что сгенерировала модель). Рамки: «попал в выдачу», «качество выдачи»,
  «объединить выдачу»; «до выдачи» ✓ только про формирование результатов поиска (ACL отсекает чанки до
  выдачи). **Запрещённая рамка** — «выдача» как передача ответа пользователю (issuance): «проверять ответ
  перед выдачей» ✗ → «перед отправкой пользователю» (иначе читается «до выдачи поиска»). Глагол «выдаёт» —
  обычное слово (модель выдаёт текст, cross-encoder выдаёт число), резервацией не связан.
- **«вывод»: output vs умозаключение.** Output-смысл — всегда с хозяином или устойчивым определением:
  «вывод модели», «сырой вывод», «структурированный вывод», «вредный вывод»; в схемах-цепочках
  («промпт → вывод → шаги агента») хозяина задаёт цепочка. Умозаключение — рамки «сделать вывод»,
  «практический вывод». Голый «вывод» вне этих рамок не писать — рядом с моделью он двоится между
  output и заключением. Результат работы инструмента — только «результат инструмента» (§1.2):
  «вывод инструмента» ✗, «ответ инструмента» ✗.
- **«конвейер» vs «пайплайн» vs «цепочка».** «Конвейер» — статическая фиксированная последовательность
  шагов, объект контраста Части I («статический конвейер» против агентного цикла). «Пайплайн» — развёрнутая
  end-to-end система, которую меряют, трейсят и настраивают (eval/observability-контекст). **Последовательная
  топология агентов (agent chain) — «цепочка», НИКОГДА «конвейер»**: конвейер закреплён за статической
  последовательностью Части I, а цепочка агентов — несколько агентов, каждый преобразует вывод предыдущего.
  Все три не взаимозаменяемы.
- **«плотный» vs «векторный».** *Dense retrieval* = «плотный поиск» (глосс термина); «векторный поиск» —
  общее описание механики (поиск по векторам). В мостах и таблицах dense глоссируется как «плотный».
- **«поиск» = retrieval по умолчанию.** Голый «поиск» в теле урока — всегда retrieval (§1.2). Другие
  смыслы живут только в связанных именах: «веб-поиск» (инструмент агента), «поиск по дереву/графу планов»
  (алгоритмический перебор, ToT), «поиск инструментов» (tool-RAG — retrieval по каталогу инструментов под
  запрос). Голым «поиском» их не называть.
- **«выбор инструмента» vs маршрутизатор.** Единственный доконтурный выбор маршрутизатора (router)
  допустимо описывать как выбор инструмента; «выбор инструмента» как активность каждого шага цикла
  сохраняет агентный смысл (tool selection).
- **«инструмент»: инструмент агента vs софт-продукты.** (1) Термин tool-use-семьи — то, что вызывает
  модель: рамки «вызов / описание / результат / выбор инструмента», «доступ к инструментам», «модель
  ходит в инструменты». (2) Продукты и фреймворки (LangSmith, Guardrails AI…) — всегда с уточнителем или
  перечнем имён: «типовые готовые инструменты» (§1.2), «инструменты наблюдаемости», «(Инструменты — X,
  Y…)». В Части II голое «инструменты» читается как инструменты агента — продуктовый смысл там без
  уточнителя запрещён. Бытовая метафора («декомпозиция становится рабочим инструментом») — обычное слово.
- **«решение»: decision vs solution.** В агентных рамках «решение» = **decision**: шаг цикла
  «рассуждение → решение → действие → наблюдение», «решение о маршруте», «модель принимает решение»,
  «решает модель». Solution-смысл — только в явной рамке «решение задачи/проблемы» и «готовое решение»;
  голое «решение» рядом с циклом читается как шаг цикла. Глагол «решает задачу» (solves) — обычное слово.
  Заголовок урока Agentic RAG («retrieval становится решением, а не шагом») — decision-смысл, признан
  допустимым как сознательная игра.
  **Формула агентного цикла** в корпусе — всегда четырёхшаговая: «рассуждение → решение → действие →
  наблюдение». Трёхшаговая «рассуждение → действие → наблюдение» допустима только там, где явно
  описывается канонический ReAct как таковой.
- **«слой»: ступень книги vs слой защиты.** Два законных смысла, разводятся рамкой. (1) Ступень
  конвейера/книги (§5): «слой Retrieval», «углубление слоя», «мостик к слою X», «слой управления над
  циклом». (2) Архитектурный/защитный слой: «защитный слой на входе и выходе», «защиту строят слоями»,
  «эшелонированная защита», «самый умный слой — рефлексия». Оба могут жить на одной странице
  (planning-loops) — но каждая фраза обязана держать рамку; голый «слой» там, где конкурируют оба чтения,
  не писать — уточняй («этот слой курса» / «слой защиты»).
- **«маршрутизация»: четыре уровня, рамка обязательна.** (1) Часть I — доконтурный выбор индекса/инструмента
  (query router); (2) Часть II — выбор инструмента в цикле (это «выбор инструмента», НЕ маршрутизация,
  §1.2); (3) Часть III — «маршрутизация запросов между моделями» (model routing); (4) Часть II,
  мультиагентные топологии — распределение подзадач между агентами: только в явной рамке («маршрутизация
  между агентами», оркестратор «декомпозиция + маршрутизация + синтез»). Голая «маршрутизация»
  допустима только внутри рамки, однозначно задающей уровень (например, в перечне функций LLM-шлюза сразу
  после определения model routing); на стыке уровней и в первом упоминании — только полная форма
  (уровень 1 — «маршрутизация запросов по индексам», как в глоссарии).
- **«батч»: GPU-планировщик vs тариф vs группа вызовов.** Смысл (1) — планирование инференс-сервера:
  «батч», «непрерывный батчинг», «работающий батч». Смысл (2) — асинхронный тариф API: только «пакетный
  режим» / «пакетный тариф», никогда «батч-тариф» — иначе рушится развод смыслов, который уроки явно
  преподают. Смысл (3) — группировка параллельных вызовов инструментов (Часть II): называем «группа» /
  «в одну группу» / «собрать в группу», НИКОГДА «батч» — «батч» зарезервирован за (1) и (2).
- **«счёт»: три смысла, рамка обязательна.** (1) Счёт-фактура / биллинг — «облачный счёт», «счёт за
  облако», «счёт за токены»; расходы в счёт «ложатся» или «попадают», деньги «падают на счёт» получателя —
  гибрид «расходы/токены падают в счёт» запрещён. (2) Учётная запись пользователя или облака — только
  «аккаунт» / «учётная запись», никогда «счёт». (3) Счётная игра слов («где считаются твои токены») —
  допустима там, где двойное чтение заякорено контекстом (закреплено за заголовком cloud-platforms).
  Голый «счёт» там, где доступно чтение «учётная запись», не писать.
- **«уровень»: тариф vs слой vs надстройка vs усилие рассуждения.** Тарифный уровень (pricing tier) —
  «уровень Reserved», «тарифный уровень» ✓. Слой абстракции — «слой» (резервация выше). Готовый
  платформенный продукт поверх эндпоинта (packaged offering) — всегда «надстройка», никогда «уровень»
  (§1.2). Степень усилия рассуждения модели (reasoning effort / `thinking_level`) — «уровень рассуждения»,
  «дискретные уровни»: законный четвёртый смысл, вводится только в рамке управления рассуждением; рядом с
  тарифами, слоями и надстройками по-прежнему запрещён без уточнителя.
- **«резерв / резервный»: fallback vs зарезервированная ёмкость.** «Резервный маршрут», «цепочка
  резервов» — fallback; «зарезервированная ёмкость», «уровень Reserved» — provisioned/reserved capacity.
  Голый «резерв» рядом с тарифами и ёмкостью не писать.
- **«маршрут»: три рамки.** (1) Маршрут веб-фреймворка (FastAPI route) — обычное слово в рамке веб-кода;
  (2) «резервный маршрут» (fallback); (3) «решение о маршруте» Части I. Каждая фраза держит свою рамку.
- **«провайдер» vs «поставщик» (vendor).** «Провайдер» — оператор API модели в эксплуатационной рамке:
  «API провайдера», «сбои у провайдера», «кэш промпта у провайдера», «дрейф модели у провайдера».
  «Поставщик» — коммерческая сторона в организационной рамке: «поставщик модели», «контракт с
  поставщиком», «привязка к поставщику (vendor lock-in)». Одна компания может быть обоими — рамка выбирает
  слово. «Вендор» — только в устойчивом «вендор-нейтральный» и в сложениях типа «мультивендорный»;
  «привязка к вендору» и «аппетиты вендоров» не писать.
- **«дрейф» = drift** (три вида: входного трафика, корпуса, модели у провайдера). Бытовой глагол
  «дрейфовать» рядом с термином не употреблять. Агентный смысл Части II — агент отклоняется от задачи —
  всегда «уход от цели», никогда «дрейф» (закреплено в глоссарной статье Reflection и уроке про
  планирование и циклы).
- **«память»: три смысла, разводятся рамкой.** (1) Параметрическая память модели — рамки «из памяти»,
  «параметрическая память», «ответила из памяти» (то, что модель знает без поданного контекста).
  (2) Память агента — всегда «рабочая память (scratchpad)» (§1.2); в углублениях — типы долговременной
  памяти «эпизодическая память», «семантическая память», «процедурная память»; во фреймворках оркестрации —
  рамка «управляемая фреймворком память (агента)», «память
  агента» (компонент фреймворка, хранящий состояние между шагами и прогонами). (3) Железо/ресурсы — только в перечнях стоимости («дороже: память, скорость поиска,
  стоимость»). Голая «память» вне этих рамок не писать.

## 3. Написание и регистр

- **top-K** — латиницей, строчная «t», в любом месте прозы и диаграмм («топ-K» запрещено). Заголовок
  словарной статьи в глоссарии — «Top-K» (правило заголовков). Русское «топ-несколько» как обычное слово —
  допустимо.
- **Recall@K / Precision@K** — с заглавной, как в глоссарии.
- **LLM-as-a-judge** — только полная форма (не «LLM-as-judge»).
- **lost-in-the-middle** — через дефисы во всех позициях.
- **nDCG, MRR, RRF, BM25, ACL, HyDE, ReAct** — фиксированное написание, как здесь.
- **AI** — латиницей во всех позициях («AI-дельта», «AI-инструменты»), никогда «ИИ». Норма корпуса
  (intro.md, подписи видео, «AI-дельта» в уроках).
- **ReAct** раскрывается как **Reasoning + Acting** (наблюдение — обратная связь цикла, не часть
  акронима).
- `retrieve → generate` — во фрагментах кода с пробелами вокруг стрелки.
- LLM — женский род («модель»): «LLM решила», «она возвращает».
- Апострофное склонение латинских терминов: bi-encoder'а, cross-encoder'ом.
- **Подписи в Mermaid-схемах:** имена протокольных артефактов остаются английскими («tool call: …»,
  «tool result: …»); обычные слова-подписи следуют языку страницы и регистру соседних уроков — с заглавной
  первой буквы («Рассуждение», «Модель», «Ответ»).
- **M×N / N+M** — без пробелов, когда это имя термина в прозе («проблема M×N интеграций», «выгода N+M»);
  с пробелами — только как арифметика в код-спане (`M × N`, `N + M`).
- **«агент» в винительном падеже — одушевлённое**: «агента подключали», «под одного агента, под другого
  агента»; «подключали агент» ✗.
- **Оси протоколов — стрелочная форма**: «агент ↔ инструменты», «агент ↔ агент» — в заголовках и
  формулировках оси (MCP против A2A). Дефисная форма «агент-к-агенту» допустима только там, где напрямую
  раскрывается аббревиатура A2A (Agent-to-Agent).
- **Ragas** — именно так (заглавная только первая буква), по текущему самоименованию проекта в его
  документации; «RAGAS» (капс из заголовка статьи 2023 года) в корпусе не писать.
- **promptfoo** — строчная «p» во всех позициях; предложение с него не начинать (перестроить фразу).
- **vLLM** — строчная «v» во всех позициях; **SGLang**, **PagedAttention**, **gpt-oss** — как здесь.
- **TTFT, SSE, PTU, SLA, IAM, OTel, LLMOps, FinOps** — фиксированное написание.
- **KV-кэш** — латинское «KV», дефис, кириллическое «кэш» (только «кэш», не «кеш»). В EN-тексте:
  существительное — «KV cache» (раздельно), атрибутив — «KV-cache» через дефис («KV-cache management»),
  по общей английской норме.
- **p50 / p95** — строчная латинская «p».
- **чек-лист** — через дефис («прод-чек-лист»).

## 4. Повторяющиеся блоки (точные строки)

- **Заголовок урока (H1) — только описательная часть, без имени из оглавления.** Имя урока (термин слоя,
  название темы) живёт во frontmatter `title:` — оттуда оно попадает в сайдбар, «хлебные крошки» и вкладку
  браузера, — поэтому повторять его в H1 не нужно: H1 несёт **описательную формулировку смысла**
  («Мультиагентные системы» в оглавлении → H1 «Когда одного агента становится мало»; «Serving» → «Из
  Jupyter-ноутбука в сервис»). Так пишут в книгах: короткое имя в содержании, раскрывающий подзаголовок на
  странице. Правило: `title:` во frontmatter НЕ трогаем (он держит навигацию); H1 = описательная часть,
  первая буква заглавная (термин, стилизованный строчным, в начале H1 получает заглавную — §3). Не «имя —
  описание» в самом H1: это дублирует крошки и удлиняет заголовок. Действует для всех уроков всех частей;
  обзоры частей («Часть I — RAG») и служебные страницы (глоссарий, интро) — исключение, там H1 = имя.
- Закрывающий раздел урока: **«## Что забрать из урока»** (EN: **"What to take away"**). На странице из
  нескольких самостоятельных разделов (как Ingestion) — «### Что забрать из раздела» в конце каждого
  раздела.
- Список терминов — последней строкой контента: **«\*\*Новые термины\*\* → [Глоссарий](…): …»** (EN: "New
  terms"). Пункт списка повторяет заголовок словарной статьи **за вычетом русского глосса в скобках**
  («Dense retrieval (плотный поиск)» → «dense retrieval»); английские скобки заголовка сохраняются
  («Attack success rate (ASR)» → «attack success rate (ASR)», «Latency (p50 / p95)» → «latency (p50 /
  p95)»); пробелы вокруг «/» — дословно как в заголовке. Регистр первой буквы — строчный, канонические
  имена слоёв и собственные имена сохраняют заглавные.
- Пометка углубления — в самом конце страницы, после `---`:
  `:::note[Дальше — углубление слоя]` + «🚧 Второй проход: …» (EN: `:::note[Next — going deeper]` +
  "🚧 Second pass: …"). 🚧 всегда означает «запланировано следующим проходом», никогда — «база не дописана».
  **Когда углубляющая страница опубликована, 🚧-пометка заменяется живым указателем** — простым, без
  маркетинга (§«Deepening / Часть 2» в CLAUDE.md): заголовок `:::note[Дальше — вторая часть урока]` (EN
  `:::note[Next — part 2 of the lesson]`) + строка «**[<описательный ярлык>](./deep-dive.md)** — <тема>»,
  называющая углубление его **описательным sidebar-ярлыком** (не «Часть 2 — Глубже»), затем «См. также:»
  с фактическими ссылками на соседние уроки (перечень-анонс — мосты не ставятся, §7). Прецеденты —
  tool-use/index («Надёжность и масштаб») и agentic-rag/index («Итеративный поиск и оценка»).
- Видео: `:::tip[▶ Видео]` (EN: `:::tip[▶ Video]`) + компонент `<YouTube id="…" title="…" />` + одна
  строка «зачем смотреть». На странице из нескольких самостоятельных разделов видео-мостик к следующему
  разделу может стоять **после** подвала «Новые термины» своего раздела (прецедент — Ingestion).
- Статусная пометка обзора части: `:::note[Статус]` с честным 🚧, пока часть растёт.
- Пометка «Предпосылки» урока: `:::note[Предпосылки]` (EN: `:::note[Prerequisites]`) — когда урок
  опирается на знание стороннего инструмента, которому книга не учит; внутри допустима ссылка на
  официальную документацию инструмента (единственное исключение из правила «внешние ссылки — только в
  глоссарии», §5).
- **Кода, закрывающая часть** (последний урок части): тематический разрыв (`---` на отдельной строке,
  пустые строки вокруг) + самостоятельный абзац-кода без заголовка — короткой коде заголовок не положен
  как приём. Действует для всех будущих закрывающих уроков части.
- **Точки входа читаются в отрыве от страницы.** Чек-листы, «Что забрать», заголовки, статьи глоссария и
  подписи схем читают вразбивку, без контекста страницы, — каждая такая строка обязана расшифровываться
  сама: голые анафоры к введённому экраном выше разворачивать («триада соответствия — резидентность,
  обязательство не обучать, частное подключение», а не просто «триада»); слова-числительные (триада, пара,
  оба, «вещи четыре») сверять счётом с перечислением, которое строка сама показывает, — «резидентность и
  соответствие: сходится ли триада…» (два названо, три заявлено) — дефект, даже если предложение гладкое.

## 6. Голос

- Уроки — **«ты»**: «держи в голове», «заэмбеддил — получил». Никогда «вы».
- Авторское **«мы»** — дозированно и только для рамки общего пути («мы наращиваем свободу модели», «мы уже
  не раз упирались») и оговорок о рамках материала; никогда — в инструкциях читателю.
- Разрешённые авторские фигуры (не «чинить»): метафорическая семья «питают/кормит», «серебряная пуля»,
  dev-жаргон «предпосчитать/предпосчёт», авторская рамка «AI-дельта», редкое оценочное суждение
  («гибрид — самое серьёзное улучшение»), «право вето» (человеческое одобрение в HITL-точке),
  «две поверхности» (авторская чеканка guardrails; вводится как «две поверхности — вход и выход»),
  «диета токенов», «сбои — не катастрофа, а рутина»,
  «регрессионные ворота (quality gate)» *(на испытании)* — фигура llmops; мост «(quality gate)» обязателен
  при первом употреблении на странице.
- Фигуры завершающего урока `real-agents` (подтверждены холодным прогоном постфактум, PR-cold-fix — при первичной
  публикации холодный прогон был пропущен из-за обрыва по 5-часовому лимиту, отсюда набор калек, вычищенных
  отдельным проходом): метафора-каркас «части тела» для разделов — «Руки» (инструменты), «Голова»
  (планирование и циклы), «Тормоза» (хуки и ограничители); «три вывески» (три вендора как витрины одного
  приёма). «Разъём» (§7) — каноническая метафора MCP из §1.2. **Не расширять фигуру за пределы заголовка:**
  «руки-инструменты» в перечне и «Руки … получает» в прозе — переигрывание; в бегущем тексте — прямые слова
  («инструменты»), метафора живёт только в заголовке раздела. **Отвергнуто:** «форма провода» (wire shape) —
  наивный читатель декодирует её как «физическую форму провода», а не как формат обмена на проводе (калька
  английского *wire format*); заменено на «форма обмена» (§1.3).
- Фигуры урока-углубления `tool-use/deep-dive` (подтверждены холодным прогоном этого PR при первом
  контакте): **«fan-out / fan-in»** (kept-EN + глосс «разветвление вызовов и сборка результатов»; наивный
  монолингвальный читатель декодирует образ из русского глосса и распаковки в предложении, а не через
  английскую страницу — в отличие от «батареек»); **«ошибка — это промпт»** (родня подтверждённой «описание
  инструмента — это промпт»); **«токенный налог»** (постоянная плата токенами за описания всех инструментов
  в каждом запросе — оба наивных читателя декодировали образ нативно при холодном прогоне этого PR).
- Фигуры урока-углубления `agentic-rag/deep-dive` (подтверждены холодным прогоном этого PR — оба наивных
  монолингвальных читателя декодировали образ при первом контакте без EN-страницы): **«дофантазирует»**
  (модель достраивает недостающее = галлюцинирует), **«набивает контекст»** (пухнет от лишних фрагментов),
  **«крутиться вхолостую» / «детерминированное кручение на месте»** (цикл повторного поиска), **«уверенная
  чушь»** (fluent-but-wrong ответ из кусков разных шагов), **«находка» / «извлечённая суть» / «дистилляция
  находки»** (перенос-вперёд дистиллята шага вместо сырых чанков; в теле сопровождается глоссом «ответ на
  подвопрос, добытый факт»), **«заход»** (hop/проход поиска). Все — родные русские слова в родном фрейме,
  не кальки.
- Фигуры урока-углубления `planning-loops/deep-dive` (подтверждены холодным прогоном этого PR — оба наивных
  монолингвальных читателя декодировали образ при первом контакте): **«пятится из тупиков»** (backtracking —
  возврат из тупика ветки), **«раскладывает по полкам»** (систематизирует типы памяти), **аналогия
  «оперативная память / диск / виртуальная память ОС»** для main/external context MemGPT (вводит образ
  виртуальной памяти явно, иначе «виртуальное управление» читается как «удалённое»). Все — родные русские
  слова в родном фрейме.
- Фигуры урока-углубления `multi-agent/deep-dive` (подтверждены холодным прогоном этого PR — оба свежих
  наивных читателя декодировали образ при первом контакте): **«конверт»** сообщения и **«более тощий конверт»**
  (схема сообщения = оболочка с полями; меньше полей — тоньше конверт), **«магнит для раздувания контекста»**
  (общая доска стягивает шум всех агентов), **«дерутся за доску»** (конкуренция за запись в общую память),
  **«сшивка траектории»** (санкционирована в §1.3 — швейный образ для СЛЕДА, не для wiring). Все — родные
  русские слова в родном фрейме.
- Антитезу «не X, а Y» использовать умеренно: это фирменный приём, но 3+ на короткую страницу — перебор.
- **Испытательный срок фигур.** Фигура, впервые введённая в каком-то PR, не может в том же PR войти в
  список разрешённых как защищённая: до явного подтверждения «наивным читателем» при первом контакте она
  числится здесь с пометкой «(на испытании)», и рецензенты продолжают её проверять наравне с обычным
  текстом. Ближайший холодный прогон или milestone-проход фигуру подтверждает или вычёркивает. Прецедент
  вычеркнутой фигуры — «батарейки» (batteries-included): образ так и не был введён в русском тексте,
  читатель декодировал его только через EN-страницу; теперь это отвергнутый вариант (§1.2, «готовая
  надстройка»).

## 7. Направление мостов (bridge rule)

- **Глоссарий**: заголовок статьи — EN (RU в скобках): «**Dense retrieval (плотный поиск)**».
- **Уроки**: в теле — канонная форма по реестру §1; мост в скобках при **первом упоминании на странице**
  (на длинной странице — освежить при первом упоминании в новом крупном разделе). Kept-EN термин получает
  русское пояснение, русский термин — английский оригинал.
- **Плотность мостов**: никогда не ставить два моста одного термина в пределах ~экрана; короткой странице
  хватает одного моста на термин. Русский термин, дословно зеркалящий оригинал (принцип наименьших
  привилегий ↔ least privilege principle), нуждается максимум в одном мосте.
- **Подпись к видео** (`:::tip[▶ Видео]`) первым упоминанием на странице **не считается** — мост ставится
  при первом определяющем упоминании в теле урока.
- **Перечень-анонс категорий** (обзорное перечисление тем страницы до их разбора) первым упоминанием тоже
  **не считается** — мост ставится при определяющем упоминании в соответствующем разделе.
- «retrieval (поиск)» с EN-первым мостом допустимо, когда retrieval выступает именем термина/слоя
  (kept-EN в списках терминов); для действия в теле урока канонная форма — «поиск» (§1.2).
- В EN-переводе мосты не нужны — там термины родные; RU-специфичные языковые правки на EN не переносятся,
  структурные и фактические — обязательны в обоих деревьях.

## 8. Полужирный (бюджет выделения)

- **Бюджет**: не больше ~0,36 полужирных спанов на строку прозы страницы (обе локали). Полужирный —
  инструмент навигации, не интонации.
- **Что выделяем**: якорь термина при первом определяющем упоминании (обычно вместе с мостом), лид
  пункта-определения в списке, редкий одиночный тезис-максиму страницы (сознательный приём).
- **Что не выделяем**: риторическое ударение посреди предложения, целые предложения и многочастные
  клаузы, повторы уже введённых терминов — в том числе в «Что забрать из урока» (итог не пере-выделяет
  то, что уже выделено в теле). Максима длиной в клаузу оформляется кавычками или курсивом, не полужирным
  (прецедент — mcp, multi-agent).
- **Исключение**: полужирные ссылки-ярлыки в TOC «Что внутри» обзоров частей — структурный приём
  оформления, в бюджет не считаются.
